# Introduction

Before Java 10, developers were required to explicitly declare the type of every local variable, even when the type was obvious from the right-hand side of the assignment.

For example:

```java
HashMap<String, List<Employee>> employeeMap =
        new HashMap<>();
```

Although the compiler already knew the type of the object being created (`HashMap<String, List<Employee>>`), developers still had to repeat the same information on the left-hand side.

This repetition became more noticeable with:

- Generic Collections
- Stream API
- Lambda Expressions
- Builder Pattern
- Fluent APIs

To reduce unnecessary boilerplate while preserving Java's static type system, Java 10 introduced **Local Variable Type Inference**, commonly referred to as **`var`**.

---

# Why Was `var` Introduced?

One of Java's greatest strengths is its **strong static type system**. However, this strength often came at the cost of verbosity.

Consider the following example.

```java
Map<String, List<Employee>> employeeMap =
        new HashMap<>();
```

Notice that the type information appears twice:

- Left-hand side → Variable declaration
- Right-hand side → Object creation

The compiler already knows that `new HashMap<>()` creates a `HashMap<String, List<Employee>>`.

Requiring developers to repeat the same type information provides no additional benefit.

Languages such as:

- Kotlin
- Scala
- C#
- Swift

already supported some form of local type inference.

The Java language designers introduced `var` to improve readability while ensuring Java remained a **statically typed language**.

---

# Problems Before Java 10

Before `var`, Java developers frequently wrote code like this:

```java
BufferedReader reader =
        Files.newBufferedReader(path);
```

```java
Optional<User> optionalUser =
        repository.findById(id);
```

```java
Map<String, List<Employee>> employees =
        new HashMap<>();
```

```java
ResponseEntity<List<EmployeeDTO>> response =
        employeeService.fetchEmployees();
```

As enterprise applications grew, these declarations became increasingly verbose.

The primary problems were:

- Repeated type declarations
- Reduced readability
- Larger codebase
- More difficult refactoring
- Increased cognitive load

---

# Design Goals

Before introducing `var`, the Java language architects established several design principles.

## Goal 1 – Reduce Boilerplate

Developers should not have to write the same type twice.

Instead of:

```java
ArrayList<Employee> employees =
        new ArrayList<>();
```

Developers should be able to write:

```java
var employees = new ArrayList<Employee>();
```

---

## Goal 2 – Preserve Static Typing

Java must **remain a statically typed language**.

This was one of the most important design requirements.

The following code should still produce a compilation error:

```java
var age = 25;

age = "Twenty Five";
```

Compilation Error:

```text
Required:
int

Found:
String
```

Once the compiler infers the type, the variable behaves exactly like an explicitly typed variable.

---

## Goal 3 – Zero Runtime Cost

The JVM should not require any modifications.

Instead,

all work should happen during compilation.

The generated bytecode should be identical to manually written code.

---

## Goal 4 – Maintain Readability

The Java team intentionally limited `var` to **local variables only**.

Why?

Because hiding field types, method parameter types, or return types would reduce API readability.

---

# Non-Goals

The Java language designers also documented what `var` should **not** do.

`var` was **never intended** to:

- Introduce dynamic typing
- Replace explicit typing everywhere
- Hide important API information
- Change JVM behavior
- Make Java similar to JavaScript

---

# What Exactly Is `var`?

One of the most common interview questions is:

> **Is `var` a new data type?**

**Answer: No.**

`var` is **not**:

- A primitive type
- A wrapper class
- A reference type
- A generic type
- An object
- A dynamically typed variable

Instead,

`var` is a **reserved type name** that instructs the compiler:

> "Infer the type of this local variable from its initializer."

For example,

```java
var name = "Guru";
```

The compiler interprets it as:

```java
String name = "Guru";
```

Similarly,

```java
var age = 25;
```

becomes

```java
int age = 25;
```

After compilation,

there is **no trace of `var`** in the generated bytecode.

---

# Is `var` a Keyword?

This question appears regularly in Java interviews.

**Answer:**

No.

`var` is a **reserved type name**, not a Java keyword.

### What does that mean?

A **keyword** has a predefined meaning everywhere in the language.

Examples:

- `class`
- `public`
- `return`
- `if`

A **reserved type name** only has special meaning when the compiler expects a type.

Example:

```java
var number = 100;
```

Here, the compiler interprets `var` as a request for type inference.

However, the following is illegal:

```java
class var {

}
```

Compilation Error.

This restriction avoids ambiguity during compilation.

---

# How Does `var` Work?

A very common misconception is that `var` somehow changes the JVM.

It does **not**.

The JVM never performs type inference.

The **Java Compiler (`javac`)** performs all the work.

The overall process is shown below.

```text
                Java Source Code

          var name = "Guru";

                     │
                     ▼

              Java Compiler (javac)

                     │

       Reads the initializer expression

                     │

       Determines the compile-time type

                     │

       Replaces `var` internally

                     │

                     ▼

             Generates Bytecode

                     │

                     ▼

                    JVM
```

The JVM executes only the generated bytecode.

It never knows that the source code originally contained `var`.

---

# Example of Type Inference

Consider the following code.

```java
var salary = 50000.0;
```

### Step 1

The compiler reads the initializer.

```java
50000.0
```

### Step 2

It determines the compile-time type.

```
double
```

### Step 3

The compiler internally rewrites the declaration as:

```java
double salary = 50000.0;
```

### Step 4

Normal bytecode is generated.

No additional instructions are introduced.

---

# Another Example

```java
var employees = new ArrayList<Employee>();
```

The compiler determines:

```
ArrayList<Employee>
```

Internally, it behaves as if you had written:

```java
ArrayList<Employee> employees =
        new ArrayList<>();
```

Again,

the generated bytecode is the same.

---

# Key Points

- `var` performs **compile-time type inference**.
- The JVM never performs inference.
- Java remains statically typed.
- The generated bytecode contains the inferred type.
- There is no runtime overhead.
- There is no performance improvement.
- `var` only changes how developers write source code.

# Compiler Internals

One of the most common misconceptions about `var` is that the **JVM** performs type inference.

**This is incorrect.**

Type inference is performed entirely by the **Java Compiler (`javac`)** during compilation.

The JVM only executes the generated bytecode and has no knowledge that `var` was ever used.

---

# Compilation Pipeline

When you compile a Java program using:

```bash
javac Example.java
```

the compiler performs several stages before generating bytecode.

```
                 Java Source Code

              var user = new User();

                      │
                      ▼

              Lexical Analysis (Lexer)

                      │
                      ▼

                  Parsing (Parser)

                      │
                      ▼

            Abstract Syntax Tree (AST)

                      │
                      ▼

             Semantic Analysis ⭐

      • Resolve symbols
      • Resolve variable types
      • Check assignments
      • Validate generics
      • Infer `var` type

                      │
                      ▼

             Bytecode Generation

                      │
                      ▼

                  Example.class

                      │
                      ▼

                     JVM
```

The important stage for `var` is **Semantic Analysis**.

---

# What Happens During Semantic Analysis?

Consider the following code.

```java
var employee = new Employee();
```

The compiler performs the following steps.

## Step 1 – Read the Initializer

The compiler examines the right-hand side.

```java
new Employee()
```

---

## Step 2 – Resolve the Expression Type

The compiler determines:

```
Employee
```

---

## Step 3 – Replace `var`

Internally, the compiler treats the code as:

```java
Employee employee = new Employee();
```

---

## Step 4 – Continue Normal Compilation

From this point onward,

the compiler completely forgets that `var` was written.

Every subsequent stage works exactly as if the developer had written the explicit type.

---

# Important Interview Point

> **`var` exists only until semantic analysis finishes.**

After semantic analysis,

there is no concept of `var`.

The compiler now has a fully typed Abstract Syntax Tree (AST).

---

# Abstract Syntax Tree (AST)

Every Java program is converted into an **Abstract Syntax Tree (AST)** before bytecode generation.

Suppose we write:

```java
var number = 100;
```

Initially the AST roughly looks like:

```
Variable Declaration

Type

var

Initializer

100
```

After semantic analysis:

```
Variable Declaration

Type

int

Initializer

100
```

Notice that `var` has disappeared.

---

# JVM Perspective

This is one of the highest-frequency interview questions.

## Does the JVM know about `var`?

**No.**

The JVM only understands:

- Bytecode
- Instructions
- Types encoded in the class file

The JVM never reads Java source code.

It never performs type inference.

It simply executes bytecode.

---

# Source Code vs Bytecode

Source code:

```java
var age = 25;
```

Compiler interpretation:

```java
int age = 25;
```

Generated bytecode is essentially equivalent to:

```
BIPUSH 25

ISTORE_1
```

There is no instruction called:

```
VAR_STORE
```

or

```
INFER_TYPE
```

Such instructions do not exist.

---

# Bytecode Analysis

Suppose we write:

```java
public class Example {

    public static void main(String[] args) {

        var age = 25;

        System.out.println(age);
    }
}
```

If we inspect the bytecode using:

```bash
javap -c Example.class
```

The output is similar to:

```
0: bipush        25
2: istore_1
3: getstatic     java/lang/System.out
6: iload_1
7: invokevirtual println
10:return
```

Notice something important.

There is absolutely no indication that the source code contained:

```java
var age = 25;
```

The JVM simply stores an integer.

---

# Why Doesn't the JVM Support `var`?

Because Java follows an important design philosophy:

> **Language evolution should not require JVM changes unless absolutely necessary.**

By implementing `var` inside the compiler,

older JVM implementations could continue executing newly compiled bytecode without modification.

This preserves one of Java's biggest strengths:

**Backward Compatibility**

# Where Can `var` Be Used?

Java intentionally restricts `var` to situations where the compiler can safely infer the type.

---

## 1. Local Variables

This is the primary use case.

```java
var employee = new Employee();
```

Compiler infers:

```java
Employee employee = new Employee();
```

---

## 2. Enhanced For Loop

```java
for (var employee : employees) {

    System.out.println(employee.getName());

}
```

Suppose

```java
employees
```

is

```java
List<Employee>
```

The compiler infers

```java
Employee
```

for each iteration.

---

## 3. Traditional For Loop

```java
for (var i = 0; i < 10; i++) {

}
```

Compiler infers

```
int
```

because

```
0
```

is an integer literal.

---

## 4. Try-With-Resources

```java
try (var reader =
        Files.newBufferedReader(path)) {

}
```

Compiler infers

```
BufferedReader
```

---

# Where `var` Cannot Be Used

The compiler must always know the type.

Whenever type inference becomes ambiguous,

Java prohibits the use of `var`.

---

# 1. Fields

Invalid:

```java
class User {

    var name = "Guru";

}
```

Compilation Error.

---

## Why?

Fields define the structure of a class.

Imagine reading this API:

```java
class User {

    var address;

}
```

What type is `address`?

You cannot know without reading its initializer.

This hurts readability.

For this reason,

field types must always be explicit.

Correct:

```java
class User {

    String name = "Guru";

}
```

---

# 2. Method Parameters

Invalid

```java
public void save(var employee) {

}
```

Compilation Error.

---

## Why?

Method parameters form part of the public API.

The caller should immediately understand the contract.

Correct

```java
public void save(Employee employee) {

}
```

---

# 3. Return Types

Invalid

```java
public var getEmployee() {

}
```

Compilation Error.

---

## Why?

Return types define what callers receive.

Imagine seeing

```java
public var calculate() {

}
```

You have no idea what the method returns.

Therefore,

return types must remain explicit.

---

# 4. Variables Without Initializers

Invalid

```java
var employee;
```

Compilation Error.

---

## Why?

The compiler has nothing to infer.

Possible types include:

- Employee
- User
- Manager
- String
- Integer

There is simply not enough information.

---

# 5. Null Initializers

Invalid

```java
var employee = null;
```

Compilation Error.

---

## Why?

The compiler determines types from expressions.

But

```
null
```

has no concrete type.

Should it infer

```
Employee?
```

or

```
User?
```

Impossible.

Therefore,

Java rejects the declaration.

---

# Important Interview Question

## Does `var` Infer the Interface or the Implementation?

Consider:

```java
var list = new ArrayList<String>();
```

Many developers answer:

```
List<String>
```

This is **incorrect**.

The inferred type is

```java
ArrayList<String>
```

because the compiler infers the **actual compile-time type of the initializer expression**, not the interface you might have chosen manually.

If you wanted:

```java
List<String>
```

you must write it explicitly.

```java
List<String> list =
        new ArrayList<>();
```

# Why Is This Important?

Suppose later you decide to replace:

```java
ArrayList
```

with

```java
LinkedList
```

Using `var`:

```java
var list =
        new ArrayList<String>();
```

must become

```java
var list =
        new LinkedList<String>();
```

because the inferred type changes.

Whereas:

```java
List<String> list =
        new ArrayList<>();
```

can simply become

```java
List<String> list =
        new LinkedList<>();
```

without changing the variable declaration.

Programming to interfaces is still considered the better design for public-facing code.

# Advanced Type Inference Rules

At first glance, `var` appears to be a simple feature:

```java
var name = "Guru";
```

However, the compiler follows several rules while performing type inference. Understanding these rules is important for interviews because many edge cases arise from them.

---

# Rule 1 – The Initializer Must Have a Definite Type

The compiler infers the type **only from the initializer expression**.

Example:

```java
var number = 100;
```

Compiler inference:

```java
int number = 100;
```

Similarly,

```java
var pi = 3.14;
```

becomes

```java
double pi = 3.14;
```

The compiler does **not** inspect future assignments.

For example,

```java
var value = 10;

value = 20;
```

This is valid because both values are `int`.

However,

```java
var value = 10;

value = 20.5;
```

Compilation fails.

```
Required:
int

Found:
double
```

---

# Rule 2 – Inference Happens Only Once

A common misconception is that Java continuously infers the variable type.

It does not.

Inference occurs exactly once:

1. Read initializer
2. Determine type
3. Replace `var`
4. Continue compilation

After this point, the variable behaves exactly like an explicitly declared variable.

Example:

```java
var language = "Java";
```

Compiler internally converts it into:

```java
String language = "Java";
```

There is no further inference after this step.

---

# Rule 3 – Type Is Determined at Compile Time

The compiler does not execute the program to determine types.

Example:

```java
var employee = createEmployee();
```

Suppose

```java
createEmployee()
```

returns

```java
Employee
```

The compiler only looks at the **method signature**, not the runtime object.

If the method returns

```java
Employee
```

then

```java
employee
```

is also of type

```java
Employee
```

even if the runtime object is actually a subclass such as `Manager`.

This demonstrates that `var` performs **compile-time** inference, not runtime inference.

---

# Rule 4 – `var` Preserves Generic Information

Consider:

```java
var users =
        new ArrayList<User>();
```

The inferred type is:

```java
ArrayList<User>
```

Notice that the generic information is retained.

The compiler does **not** infer:

```java
ArrayList
```

or

```java
Object
```

Generic type arguments are preserved exactly as they appear in the initializer.

---

# Rule 5 – `var` Does Not Ignore Java's Type System

Example:

```java
var number = 10;

number = "Hello";
```

Compilation Error.

The compiler has already inferred:

```java
int
```

The assignment of a `String` violates Java's static type rules.

---

# Rule 6 – Inference Uses the Declared Return Type

Consider the following example.

```java
Employee createEmployee() {

    return new Manager();

}
```

Now,

```java
var employee = createEmployee();
```

Many beginners think the compiler infers:

```java
Manager
```

This is incorrect.

The inferred type is:

```java
Employee
```

because the compiler only considers the **declared return type**.

The actual runtime object is irrelevant during compilation.

---

# `var` with Generics

One of the biggest benefits of `var` is simplifying generic declarations.

Before Java 10:

```java
HashMap<String, List<Employee>> employeeMap =
        new HashMap<>();
```

After Java 10:

```java
var employeeMap =
        new HashMap<String, List<Employee>>();
```

The compiler infers:

```java
HashMap<String, List<Employee>>
```

Notice that all generic information is retained.

---

# `var` with the Diamond Operator

Example:

```java
var list = new ArrayList<>();
```

Will this compile?

**No.**

Why?

Because the compiler has no information about the generic type.

Should it infer:

```java
ArrayList<String>
```

or

```java
ArrayList<Employee>
```

or

```java
ArrayList<Integer>
```

There is no context available.

The compiler therefore infers:

```java
ArrayList<Object>
```

which is usually **not** what developers expect.

### Preferred Approach

```java
var employees =
        new ArrayList<Employee>();
```

or

```java
List<Employee> employees =
        new ArrayList<>();
```

Both are better than relying on an empty diamond with `var`.

---

# `var` with Anonymous Classes

Consider:

```java
var object = new Object() {

    void print() {

        System.out.println("Hello");

    }

};
```

This is perfectly valid.

The inferred type is the **anonymous class type** itself.

This allows you to access methods declared within that anonymous class.

Example:

```java
object.print();
```

works successfully.

If the variable were declared as:

```java
Object object =
        new Object() {

    void print() {

    }

};
```

then

```java
object.print();
```

would fail because `Object` has no such method.

---

# `var` with Lambda Expressions

Consider:

```java
var operation =
        (a, b) -> a + b;
```

Compilation Error.

---

## Why?

A lambda expression has **no standalone type**.

A lambda requires a **target type**.

For example:

```java
Function<Integer, Integer> square =
        x -> x * x;
```

The compiler knows the lambda implements:

```java
Function<Integer, Integer>
```

Without this target type,

the compiler cannot determine which functional interface the lambda belongs to.

Therefore,

`var` cannot infer the type here.

---

# `var` Inside Lambda Parameters

Java 11 introduced support for `var` inside lambda parameters.

Example:

```java
(var x, var y) -> x + y
```

Why was this added?

Primarily to support annotations.

Example:

```java
(@Nonnull var name) -> name.length()
```

Notice that **every parameter** must use `var`.

The following is illegal:

```java
(var x, int y) -> x + y
```

Parameter declarations inside a lambda must be consistent.

---

# Advantages of `var`

## 1. Reduces Boilerplate

Instead of writing:

```java
HashMap<String, List<Employee>> employees =
        new HashMap<>();
```

write:

```java
var employees =
        new HashMap<String, List<Employee>>();
```

The intent remains clear while reducing repetition.

---

## 2. Improves Readability

When the type is obvious,

`var` makes code cleaner.

Example:

```java
var customer = new Customer();
```

The type is immediately apparent.

---

## 3. Simplifies Stream API Code

Without `var`:

```java
List<String> names =
        employees.stream()
                 .map(Employee::getName)
                 .toList();
```

With `var`:

```java
var names =
        employees.stream()
                 .map(Employee::getName)
                 .toList();
```

The focus shifts to the business logic instead of the declaration.

---

## 4. Easier Refactoring

Suppose the constructor changes from

```java
HashMap
```

to

```java
ConcurrentHashMap
```

With `var`, only the initializer changes.

---

# Disadvantages of `var`

## 1. Reduced Readability

Example:

```java
var result = calculate();
```

Without inspecting

```java
calculate()
```

the reader has no idea what `result` represents.

---

## 2. Encourages Programming to Implementations

Example:

```java
var list =
        new ArrayList<String>();
```

The inferred type is:

```java
ArrayList<String>
```

not

```java
List<String>
```

This can make future implementation changes more difficult.

---

## 3. Easy to Overuse

Some developers write:

```java
var a = foo();

var b = bar();

var c = baz();
```

Although legal,

the code becomes much harder to understand.

Remember:

> Less typing should never come at the cost of readability.

---

# Best Practices

## ✅ Use `var` When the Type Is Obvious

Good:

```java
var employee = new Employee();
```

---

## ✅ Use `var` with Streams

```java
var names =
        employees.stream()
                 .map(Employee::getName)
                 .toList();
```

---

## ✅ Use `var` with Fluent APIs

```java
var response =
        webClient.get()
                 .retrieve()
                 .bodyToMono(User.class);
```

---

## ❌ Avoid `var` for Complex Return Types

Poor:

```java
var data = calculateStatistics();
```

Better:

```java
Statistics statistics =
        calculateStatistics();
```

The explicit type improves readability.

---

## ❌ Do Not Use `var` Everywhere

Treat `var` as a readability feature,

not a typing shortcut.

A good rule is:

> If a developer cannot determine the type within a few seconds, use an explicit type instead.

Incorrect assumption:


Some developers expect this to work because languages like JavaScript allow variables to change types.

Compilation Error:

```

Found:
String
```

### Explanation

```java
int value = 10;
```

## Interview Tip

> **`var` reduces typing, not type safety.**

# Mistake 2 – Using `var` When the Type Is Not Obvious

Consider the following code.

```java
var result = process();
```

Can you answer these questions immediately?

- What is the type?
- Is it a `String`?
- An `Optional<User>`?
- A `ResponseEntity<User>`?
- A `List<Employee>`?

You cannot know without opening the `process()` method.

A better approach is:

```java
EmployeeReport report = process();
```

Now the code is self-documenting.

---

# Mistake 3 – Using `var` Everywhere

Poor example:

```java
var a = calculateA();

var b = calculateB();

var c = calculateC();

var d = calculateD();
```

Nothing in this code tells the reader what each variable represents.

A balanced approach is:

```java
Customer customer = customerService.findById(id);

var orders = orderRepository.findByCustomer(customer);
```

The first variable benefits from an explicit type.

The second is obvious because of the repository method name.

---

# Mistake 4 – Forgetting That `var` Infers the Implementation

Example:

```java
var list = new ArrayList<String>();
```

The inferred type is:

```java
ArrayList<String>
```

Many developers mistakenly believe it becomes:

```java
List<String>
```

It does not.

---

## Why Does This Matter?

Suppose you later decide to replace:

```java
ArrayList
```

with

```java
LinkedList
```

With `var`:

```java
var list = new LinkedList<String>();
```

The variable type changes automatically.

If your code depends on `ArrayList`-specific methods, this may introduce compilation errors.

Programming to interfaces remains the preferred design principle.

---

# Mistake 5 – Expecting `var` to Improve Performance

Some developers believe less typing means faster execution.

This is false.

Consider:

```java
var employee = new Employee();
```

Compiler output:

```java
Employee employee = new Employee();
```

Generated bytecode is effectively identical.

Therefore,

- No memory savings
- No CPU improvement
- No JVM optimization

The benefit is purely improved source code readability.

---

# Mistake 6 – Trying to Use `var` Without an Initializer

Example:

```java
var employee;
```

Compilation Error.

The compiler cannot infer a type because there is no initializer expression.

Always remember:

> **No initializer → No inference.**

---

# Mistake 7 – Initializing with `null`

Example:

```java
var user = null;
```

Compilation Error.

Why?

Because `null` has no type.

The compiler cannot determine whether the variable should become:

- User
- Employee
- String
- Integer
- Object

---

# Performance Analysis

One interview question appears repeatedly.

> **Does `var` improve performance?**

The answer is **No**.

---

# Compilation Comparison

Example 1

```java
Employee employee = new Employee();
```

Example 2

```java
var employee = new Employee();
```

After compilation,

both produce essentially the same bytecode.

The JVM executes both programs identically.

---

# Memory Consumption

Exactly the same.

Both allocate:

```
Employee Object
```

on the heap.

The variable itself stores a reference.

There is no additional memory used because of `var`.

---

# Execution Speed

Exactly the same.

Since the generated bytecode is equivalent,

execution time is identical.

---

# Garbage Collection

No impact.

Garbage Collection depends on object reachability,

not on whether the source code used `var`.

---

# Spring Boot Examples

---

## Example 1 – Repository

```java
var user =
        userRepository.findById(id);
```

Assuming the repository method returns:

```java
Optional<User>
```

the inferred type becomes:

```java
Optional<User>
```

The method name clearly communicates the return type, making `var` a good choice.

---

## Example 2 – Service Layer

```java
var dto =
        employeeMapper.toDto(employee);
```

The mapper method clearly indicates that a DTO is returned.

Using `var` improves readability by reducing repetition.

---

## Example 3 – REST Controller

```java
@GetMapping("/{id}")
public ResponseEntity<UserDTO> getUser(
        @PathVariable Long id) {

    var user = userService.findById(id);

    return ResponseEntity.ok(user);
}
```

Notice that only the **local variable** uses `var`.

The method signature remains explicit.

This is considered a good practice.

---

## Example 4 – Stream API

```java
var names =
        employees.stream()
                 .map(Employee::getName)
                 .toList();
```

Without `var`, the declaration becomes longer while adding little value.

---

## Example 5 – WebClient

```java
var response =
        webClient.get()
                 .retrieve()
                 .bodyToMono(User.class);
```

Fluent APIs are one of the best use cases for `var`.

# Interview Questions and Answers

---

## Q1. What is `var`?

### Answer

`var` is a Java 10 feature (JEP 286) that enables **local variable type inference**.

The compiler determines the variable's type from its initializer during compilation.

It is **not** a new data type.

---

## Q2. Does `var` make Java dynamically typed?

### Answer

No.

Java remains statically typed.

Once the compiler infers the type, it cannot change.

---

## Q3. Is `var` a keyword?

### Answer

No.

It is a **reserved type name**, not a Java keyword.

---

## Q4. Can `var` be used for fields?

### Answer

No.

Fields are part of a class's public structure and must have explicit types.

---

## Q5. Can `var` be used for method parameters?

### Answer

No.

Method parameters define the method contract and must be explicitly typed.

---

## Q6. Can `var` be used for return types?

### Answer

No.

Return types are part of the public API and must remain explicit.

---

## Q7. Why can't we write `var value = null;`?

### Answer

Because `null` has no concrete type.

The compiler has nothing from which to infer the variable type.

---

## Q8. Does `var` improve performance?

### Answer

No.

The generated bytecode is effectively identical to explicitly typed code.

There is no runtime improvement.

---

## Q9. Does the JVM know about `var`?

### Answer

No.

The JVM executes bytecode.

Only the compiler (`javac`) processes `var`.

---

## Q10. Does `var` preserve generic types?

### Answer

Yes.

Example:

```java
var users = new ArrayList<User>();
```

The inferred type is:

```java
ArrayList<User>
```

Generic information is preserved during compilation.

---

## Q11. What type is inferred here?

```java
var list = new ArrayList<String>();
```

### Answer

The inferred type is:

```java
ArrayList<String>
```

**Not**

```java
List<String>
```

The compiler always infers the actual compile-time type of the initializer expression.

---

## Q12. Should `var` be used everywhere?

### Answer

No.

Use `var` only when it improves readability.

If the type is not obvious from the initializer, use an explicit type.

---

# Quick Revision Notes

- Introduced in **Java 10 (JEP 286)**.
- Provides **local variable type inference**.
- Implemented entirely by the **compiler (`javac`)**.
- The JVM never sees `var`.
- Java remains **statically typed**.
- Type inference occurs **once**, during compilation.
- `var` requires an initializer.
- `var` cannot be used for:
    - Fields
    - Method parameters
    - Return types
    - Variables initialized with `null`
- The inferred type is the **actual compile-time type** of the initializer.
- There is **no runtime overhead** and **no performance improvement**.
- Use `var` when the type is obvious and readability improves.

# Chapter Summary

`var` is a **compile-time language enhancement** introduced in Java 10 to reduce boilerplate by enabling local variable type inference. The feature is implemented entirely by the Java compiler and has **no impact on the JVM, memory usage, or runtime performance**. It preserves Java's static type system while improving code readability when used appropriately. The key to using `var` effectively is balancing conciseness with clarity—use it where the inferred type is obvious and avoid it where it hides important information.

# Chapter 2 – Records (Java 16)

> **Java Version:** Java 16 (Final)
>
> **Preview:** Java 14 & Java 15
>
> **JEP:** 395 – Records
>
> **Category:** Language Enhancement
>
> **Interview Difficulty:** ⭐⭐⭐⭐☆ (Frequently Asked in Java 17+ Interviews)

---

# Introduction

One of the biggest criticisms of Java has always been the amount of **boilerplate code** required to create simple data-holding classes.

In enterprise applications, developers frequently create classes whose primary purpose is to **transfer data** between different layers of the application.

Examples include:

- Request DTOs
- Response DTOs
- Configuration Objects
- Event Payloads
- Kafka Messages
- RabbitMQ Messages
- Projection Objects
- Value Objects

Most of these classes contain very little business logic.

Instead, they repeatedly define:

- Fields
- Constructors
- Getters
- `equals()`
- `hashCode()`
- `toString()`

This repetitive code adds little business value and increases maintenance effort.

To solve this problem, Java introduced **Records** in Java 16.

A Record allows developers to describe **what data an object contains**, while the compiler automatically generates the repetitive code.

---

# Why Were Records Introduced?

Consider a simple DTO representing a user.

Before Java 16, it looked like this:

```java
public final class UserDTO {

    private final String name;
    private final String email;
    private final int age;

    public UserDTO(String name,
                   String email,
                   int age) {

        this.name = name;
        this.email = email;
        this.age = age;
    }

    public String getName() {
        return name;
    }

    public String getEmail() {
        return email;
    }

    public int getAge() {
        return age;
    }

    @Override
    public boolean equals(Object obj) {

    }

    @Override
    public int hashCode() {

    }

    @Override
    public String toString() {

    }

}
```

Although this class contains only **three pieces of business data**, the source file grows to nearly **50 lines**.

Most of this code is repetitive and identical across hundreds of DTOs.

Records eliminate this duplication.

The same class becomes:

```java
public record UserDTO(
        String name,
        String email,
        int age
) {}
```

The compiler generates everything else automatically.

---

# Problems Before Records

Before Java 16, Java developers encountered several recurring issues.

## Problem 1 – Boilerplate Code

Every DTO required:

- Fields
- Constructor
- Getter methods
- `equals()`
- `hashCode()`
- `toString()`

Developers spent more time writing infrastructure code than business logic.

---

## Problem 2 – Error-Prone Implementations

Imagine manually implementing `equals()`.

```java
@Override
public boolean equals(Object obj) {

    ...
}
```

Forgetting even one field could introduce subtle bugs.

Similarly,

incorrect implementations of `hashCode()` could break collections like:

- `HashMap`
- `HashSet`
- `ConcurrentHashMap`

---

## Problem 3 – Mutable DTOs

Many DTOs accidentally exposed setter methods.

```java
user.setName("Alex");
```

Mutable objects can easily introduce bugs when shared between threads or application layers.

Records encourage immutable design by default.

---

## Problem 4 – Maintenance

Suppose a new field is added.

```java
private String phoneNumber;
```

Developers also had to update:

- Constructor
- Getters
- `equals()`
- `hashCode()`
- `toString()`

Forgetting one update could create inconsistent behavior.

---

# Design Goals

Records were designed with several objectives.

## Goal 1 – Reduce Boilerplate

Developers should declare only the state.

The compiler should generate repetitive code automatically.

---

## Goal 2 – Encourage Immutability

Data transfer objects should be immutable by default.

Immutable objects are:

- Thread-safe
- Easier to reason about
- Less error-prone
- Safer for concurrent programming

---

## Goal 3 – Preserve Java's Object-Oriented Model

Records should remain normal Java classes.

They should:

- Be instantiated using `new`
- Participate in polymorphism
- Implement interfaces
- Support methods
- Work with existing libraries

---

## Goal 4 – Improve Readability

Instead of reading fifty lines of boilerplate,

developers should immediately understand the important part:

```java
public record UserDTO(
        String name,
        String email,
        int age
) {}
```

The entire data model becomes visible in a single declaration.

---

# Non-Goals

One common interview mistake is assuming Records replace normal classes.

They do not.

Records were **not** designed for:

- Mutable objects
- Rich domain models
- Complex inheritance hierarchies
- JPA entities
- Stateful business objects

Normal classes continue to be the correct choice for these scenarios.

---

# What Is a Record?

A **Record** is a **special kind of class** whose primary purpose is to model immutable data.

Unlike traditional classes,

a Record describes:

> **What data an object contains**

instead of

> **How that data should be implemented**

Example:

```java
public record Employee(

        Long id,
        String name,
        String department

) {}
```

This declaration tells the compiler:

> "Generate a complete immutable class representing these three components."

---

# Is a Record a Class?

This is one of the highest-frequency interview questions.

## Answer

Yes.

A Record is a **real Java class**.

It behaves like every other class in the JVM.

The compiler simply generates additional members automatically.

The following declaration:

```java
public record Employee(

        Long id,
        String name

) {}
```

is conceptually similar to:

```java
public final class Employee
        extends Record {

    private final Long id;

    private final String name;

    public Employee(Long id,
                    String name) {

        this.id = id;
        this.name = name;

    }

    public Long id() {

        return id;

    }

    public String name() {

        return name;

    }

    @Override
    public boolean equals(Object obj) {

    }

    @Override
    public int hashCode() {

    }

    @Override
    public String toString() {

    }

}
```

Notice something important.

You never write this code.

The **compiler** generates it.

---

# Record Architecture

```
                 Java Source

record Employee(Long id, String name)

                     │
                     ▼

                Java Compiler

                     │

       Generates Automatically

      • private final fields
      • constructor
      • accessor methods
      • equals()
      • hashCode()
      • toString()

                     │
                     ▼

              Employee.class

                     │
                     ▼

                    JVM
```

This architecture is very similar to `var`.

The compiler performs all transformations.

The JVM executes the generated class without any special treatment.

---

# Records vs Traditional Classes

| Traditional Class | Record |
|-------------------|--------|
| Manual fields | Generated |
| Manual constructor | Generated |
| Manual getters | Generated accessors |
| Manual `equals()` | Generated |
| Manual `hashCode()` | Generated |
| Manual `toString()` | Generated |
| Usually mutable | Immutable by default |
| Large amount of code | Very concise |

---

# Record Components

Consider the following Record.

```java
public record Employee(

        Long id,
        String name,
        double salary

) {}
```

The values inside the parentheses are called **Record Components**.

```
Long id

String name

double salary
```

These are **not merely constructor parameters**.

Each component simultaneously defines:

- A private final field
- A constructor parameter
- An accessor method
- Participation in `equals()`
- Participation in `hashCode()`
- Participation in `toString()`

This is one of the most important concepts to understand.

A Record Component is a **single declaration that drives multiple compiler-generated members**.

---

# Visualizing Record Components

```
Record Component

        String name

             │
             │
 ┌───────────┼────────────┐
 │           │            │
 ▼           ▼            ▼

Field   Constructor   Accessor

             │
             ▼

equals()

hashCode()

toString()
```

One declaration results in multiple generated members.

This is why changing a Record Component automatically affects several parts of the generated class.

---

# Key Points

- Records were finalized in **Java 16**.
- Records are **special classes**, not a new data type.
- Records primarily represent immutable data.
- Records reduce boilerplate significantly.
- Record Components define the complete state of the object.
- The Java compiler generates most of the implementation.
- The JVM executes Records like ordinary classes.
- Records improve readability and maintainability, especially for DTOs.

---

---

# Compiler-Generated Members

One of the biggest advantages of Records is that the **Java compiler generates most of the repetitive code automatically**.

When the compiler encounters a Record declaration, it generates several members behind the scenes.

These generated members make a Record behave like a complete Java class while allowing developers to write only a single line of code.

For example,

```java
public record Employee(

        Long id,
        String name,
        double salary

) {}
```

Although the source code is only a few lines long, the compiler generates:

- Private final fields
- Canonical constructor
- Accessor methods
- `equals()`
- `hashCode()`
- `toString()`

Let's understand each of these in detail.

---

# 1. Private Final Fields

Consider the following Record.

```java
public record Employee(

        Long id,
        String name,
        double salary

) {}
```

The compiler generates the following fields.

```java
private final Long id;

private final String name;

private final double salary;
```

Notice two important characteristics.

## Characteristic 1 – Private

The fields are private.

They cannot be accessed directly.

The following code is illegal.

```java
Employee employee =
        new Employee(1L, "Guru", 90000);

employee.name;
```

Compilation Error.

Developers must use accessor methods.

---

## Characteristic 2 – Final

Every field is declared `final`.

Once initialized,

it cannot point to another object.

Example:

```java
Employee employee =
        new Employee(1L, "Guru", 90000);
```

The following is impossible.

```java
employee.name = "Alex";
```

Compilation Error.

This is one of the primary reasons Records are immutable.

---

# Memory Layout

A Record object occupies memory just like any other Java object.

```
               Heap

        Employee Object

 ┌───────────────────────────┐
 │ Header                    │
 ├───────────────────────────┤
 │ id ───────────────► 1L     │
 ├───────────────────────────┤
 │ name ─────────────► Guru   │
 ├───────────────────────────┤
 │ salary ──────────►90000.0  │
 └───────────────────────────┘
```

There is **no special memory layout** for Records.

They occupy heap memory exactly like normal objects.

---

# Are Records Immutable?

Interviewers almost always ask this question.

The answer is:

**Yes—but with an important clarification.**

Records provide **shallow immutability**, not deep immutability.

We'll discuss this in detail later.

---

# 2. Canonical Constructor

Every Record has a **Canonical Constructor**.

A canonical constructor is a constructor whose parameters exactly match the Record Components.

Given:

```java
public record Employee(

        Long id,
        String name,
        double salary

) {}
```

the compiler generates:

```java
public Employee(

        Long id,
        String name,
        double salary

) {

    this.id = id;
    this.name = name;
    this.salary = salary;

}
```

This constructor initializes every component.

---

# Why Is It Called "Canonical"?

The word **canonical** means:

> The standard or official representation.

Since a Record is completely defined by its components,

there is exactly one constructor that represents the entire state.

That constructor is called the **Canonical Constructor**.

---

# Constructor Invocation

Creating a Record object looks exactly like creating a normal class.

```java
Employee employee =
        new Employee(

                101L,
                "Guru",
                95000

        );
```

Internally,

the generated constructor executes.

```
Allocate Memory

        │

        ▼

Initialize Fields

        │

        ▼

Return Reference
```

The object creation process is identical to ordinary Java classes.

---

# Can We Write Our Own Canonical Constructor?

Yes.

Developers can replace the generated constructor with their own implementation.

Example:

```java
public record Employee(

        Long id,
        String name

) {

    public Employee(

            Long id,
            String name

    ) {

        this.id = id;
        this.name = name;

    }

}
```

Although this is legal,

it is rarely necessary unless validation or preprocessing is required.

We'll cover constructor customization in the next chapter.

---

# 3. Accessor Methods

One of the most surprising features of Records is that they do **not generate getters**.

Instead,

they generate **Accessor Methods**.

Example:

```java
public record Employee(

        Long id,
        String name

) {}
```

Compiler generates:

```java
public Long id() {

    return id;

}

public String name() {

    return name;

}
```

Notice the names.

```
id()

name()
```

Not

```
getId()

getName()
```

This difference is intentional.

---

# Why Not Getters?

Traditional JavaBeans follow this naming convention:

```java
getName()

getAge()

getSalary()
```

Records intentionally avoid JavaBean conventions.

Why?

Because Record Components already describe the data.

The accessor method simply returns that component.

Therefore,

```java
employee.name()
```

reads naturally.

```
Employee's name
```

instead of

```
Get employee's name
```

---

# Accessor Example

```java
Employee employee =
        new Employee(

                100L,
                "Guru"

        );
```

Access values:

```java
System.out.println(employee.id());

System.out.println(employee.name());
```

Output:

```
100

Guru
```

---

# Can We Override Accessor Methods?

Yes.

Example:

```java
public record Employee(

        String name

) {

    @Override
    public String name() {

        return name.toUpperCase();

    }

}
```

Now,

```java
Employee employee =
        new Employee("Guru");

System.out.println(employee.name());
```

Output:

```
GURU
```

Although possible,

this should be used carefully because developers generally expect accessors to simply return stored values.

---

# 4. Generated equals()

Another important compiler-generated member is `equals()`.

Suppose we create two Records.

```java
Employee e1 =
        new Employee(

                1L,
                "Guru",
                90000

        );

Employee e2 =
        new Employee(

                1L,
                "Guru",
                90000

        );
```

Calling

```java
System.out.println(e1.equals(e2));
```

prints:

```
true
```

Why?

Because the compiler compares every Record Component.

Conceptually,

the generated implementation behaves like:

```java
@Override
public boolean equals(Object obj) {

    if (this == obj)
        return true;

    if (!(obj instanceof Employee other))
        return false;

    return Objects.equals(id, other.id)
            &&
           Objects.equals(name, other.name)
            &&
           salary == other.salary;

}
```

You never write this yourself.

The compiler generates it.

---

# Why Is equals() Important?

Methods like:

```java
HashSet

HashMap

ConcurrentHashMap
```

depend on correctly implemented `equals()`.

With Records,

developers no longer need to manually maintain these methods.

---

# 5. Generated hashCode()

Whenever `equals()` is generated,

the compiler also generates `hashCode()`.

This ensures the Java contract is preserved.

Recall the contract:

> If two objects are equal, they **must** have the same hash code.

The generated implementation conceptually behaves like:

```java
@Override
public int hashCode() {

    return Objects.hash(

            id,
            name,
            salary

    );

}
```

Again,

the compiler generates this automatically.

---

# Why Is hashCode() Important?

Collections such as:

- HashMap
- HashSet
- LinkedHashMap
- ConcurrentHashMap

rely on efficient hash code computation.

Records work seamlessly with these collections without additional code.

Example:

```java
Set<Employee> employees =
        new HashSet<>();

employees.add(
        new Employee(
                1L,
                "Guru",
                90000
        )
);

employees.add(
        new Employee(
                1L,
                "Guru",
                90000
        )
);

System.out.println(employees.size());
```

Output:

```
1
```

The duplicate is removed because `equals()` and `hashCode()` are generated correctly.

---

# 6. Generated toString()

The compiler also generates a readable `toString()` implementation.

Example:

```java
Employee employee =
        new Employee(

                101L,
                "Guru",
                90000

        );

System.out.println(employee);
```

Output:

```
Employee[id=101, name=Guru, salary=90000.0]
```

This is extremely useful for:

- Logging
- Debugging
- Unit Testing
- Console Output

---

# Benefits of Compiler-Generated Members

By generating these members automatically, Records provide:

- Less boilerplate code
- Consistent implementations
- Fewer human errors
- Easier maintenance
- Better readability
- Correct `equals()`/`hashCode()` implementations by default

---

# Key Takeaways

- Every Record Component becomes a **private final field**.
- Every Record receives a **canonical constructor**.
- Records generate **accessor methods**, not getters.
- The compiler automatically generates `equals()`, `hashCode()`, and `toString()`.
- These generated members make Records ideal for immutable data carriers.
- Record objects have the same heap memory layout as ordinary Java objects.
- The compiler—not the JVM—creates these members during compilation.

---

# `java.lang.Record`

One of the most frequently asked interview questions is:

> **What does a Record actually extend?**

Consider the following declaration.

```java
public record Employee(

        Long id,
        String name

) {}
```

At first glance, it appears that the Record does not extend anything.

However, every Record **implicitly extends**:

```java
java.lang.Record
```

Conceptually, the compiler treats it as:

```java
public final class Employee
        extends Record {

    ...

}
```

Unlike ordinary classes, you never write:

```java
extends Record
```

The compiler inserts it automatically.

---

# Why Was `java.lang.Record` Introduced?

Before Java 16, every class implicitly extended:

```java
java.lang.Object
```

```
Object

   ▲

Employee
```

After Records were introduced, Java needed a way to distinguish between:

- Normal classes
- Record classes

Therefore, the JDK introduced a new superclass:

```java
java.lang.Record
```

Now the hierarchy becomes:

```
Object

   ▲

Record

   ▲

Employee
```

This enables the JVM and Reflection API to identify Record classes.

---

# Record Inheritance Hierarchy

```
               Object
                  ▲
                  │
            java.lang.Record
                  ▲
                  │
        ┌─────────┴─────────┐
        │                   │
   Employee            Customer
      Record              Record
```

Every Record shares the same parent class.

---

# Can a Record Extend Another Class?

This is one of the highest-frequency interview questions.

## Answer

**No.**

Example:

```java
class Person {

}
```

Attempting:

```java
public record Employee(
        String name
)
extends Person {

}
```

Compilation Error.

---

# Why?

Java supports **single inheritance**.

A Record already extends:

```java
java.lang.Record
```

Since Java allows extending only one class,

there is no room to extend another class.

```
Object
   ▲
Record
   ▲
Employee
```

Trying to insert another superclass would violate Java's inheritance model.

---

# Can a Class Extend a Record?

Again,

the answer is:

**No.**

Suppose we have:

```java
public record Employee(
        String name
) {}
```

Attempting:

```java
class Manager
        extends Employee {

}
```

Compilation Error.

---

# Why?

Because every Record is implicitly:

```java
final
```

Final classes cannot be subclassed.

---

# Are Records Final?

Yes.

Every Record is implicitly declared as:

```java
final
```

Even if you never write it.

Conceptually,

```java
public record Employee(
        String name
) {}
```

becomes

```java
public final class Employee
        extends Record {

}
```

---

# Why Are Records Final?

The primary reason is **immutability**.

Imagine if Records could be extended.

```java
record Employee(String name) {}
```

Now suppose:

```java
class Manager extends Employee {

    String department;

}
```

Should `equals()` compare:

- only `name`
- or both `name` and `department`

The automatically generated implementation becomes ambiguous.

To avoid these problems,

Java prohibits Record inheritance.

---

# Can Records Be Abstract?

No.

The following is illegal.

```java
public abstract record Employee(

        String name

) {}
```

Compilation Error.

---

# Why?

Records represent complete data models.

A Record must always describe its full state.

Abstract Records would contradict that design principle.

---

# Can Records Implement Interfaces?

Yes.

This is one of the biggest strengths of Records.

Example:

```java
interface Printable {

    void print();

}
```

Now:

```java
public record Employee(

        String name

) implements Printable {

    @Override
    public void print() {

        System.out.println(name);

    }

}
```

This compiles successfully.

---

# Why Is This Allowed?

Interfaces define behavior,

not state.

Since Records already describe state,

implementing behavior through interfaces fits perfectly within Java's object-oriented model.

---

# Records and Polymorphism

Records fully support polymorphism through interfaces.

Example:

```java
interface Shape {

    double area();

}
```

Now define two Records.

```java
public record Circle(
        double radius

    @Override
    public double area() {

        return Math.PI * radius * radius;

    }

}
```

```java
public record Rectangle(
        double width,
        double height
) implements Shape {

    @Override
    public double area() {

    }

}
```

Usage:

```java
Shape shape =
        new Circle(10);

System.out.println(shape.area());
```

Output:

```
314.159...
```

Although Records cannot participate in class inheritance,

they fully support interface-based polymorphism.

---

- Overridden methods

Yes.

# Can Records Declare Additional Fields?

This question often appears in interviews.

Suppose we try:


Compilation Error.

---

# Why?

The state of a Record is defined **only** by its Record Components.

Allowing additional instance fields would make the generated methods inconsistent.

For example,

Should `equals()` compare:

```
name
```

or

```
name + age
```

To avoid ambiguity,

additional instance fields are prohibited.

Yes.

Only **instance fields** are forbidden.

Static fields belong to the class,

not individual objects.

Example:

```java

```

This is perfectly valid.

# Constructors in Records

Constructors are one of the most important Record topics in Java interviews.

Many developers know that Records automatically generate constructors, but very few understand:

- What type of constructor is generated
- How constructors work internally
- When to customize them
- Validation strategies
- Constructor execution order
- Compact constructors
- Defensive copying

Understanding these concepts is essential for Java 17/21 backend interviews.

---

# Types of Constructors in Records

A Record can have three constructor forms.

```
                    Record

                      │

      ┌───────────────┼───────────────┐

      ▼               ▼               ▼

Canonical      Compact        Custom (Overloaded)

Constructor    Constructor     Constructor
```

Let's examine each one in detail.

---

# 1. Canonical Constructor

Every Record automatically receives a **Canonical Constructor**.

This constructor accepts **exactly one parameter for every Record Component**, in the same order.

Example:

```java
public record Employee(

        Long id,
        String name,
        double salary

) {}
```

The compiler generates:

```java
public Employee(

        Long id,
        String name,
        double salary

) {

    this.id = id;
    this.name = name;
    this.salary = salary;

}
```

You never see this code because the compiler generates it automatically.

---

# Why Is It Called Canonical?

The term **canonical** means:

> The official or standard representation.

A Record's state is completely defined by its components.

Therefore, the constructor that initializes all components is considered the **canonical** representation of the object.

---

# Constructor Invocation

Creating a Record object:

```java
Employee employee =
        new Employee(

                101L,
                "Guru",
                90000

        );
```

Internally:

```
Allocate Heap Memory

        │

        ▼

Invoke Constructor

        │

        ▼

Assign Fields

        │

        ▼

Return Object Reference
```

The object creation lifecycle is identical to ordinary Java classes.

---

# Can We Write Our Own Canonical Constructor?

Yes.

Developers may replace the generated constructor.

Example:

```java
public record Employee(

        Long id,
        String name

) {

    public Employee(

            Long id,
            String name

    ) {

        this.id = id;
        this.name = name;

    }

}
```

Once you define this constructor,

the compiler **does not** generate another canonical constructor.

---

# Why Customize the Canonical Constructor?

Usually for:

- Validation
- Normalization
- Logging
- Defensive copying

Example:

```java
public Employee(

        Long id,
        String name

) {

    if (id == null) {

        throw new IllegalArgumentException(
                "Id cannot be null"
        );

    }

    this.id = id;
    this.name = name;

}
```

---

# Important Rule

Every Record Component **must** be assigned.

This is illegal:

```java
public Employee(

        Long id,
        String name

) {

    this.id = id;

}
```

Compilation Error.

```
Variable name might not have been initialized.
```

Every component must receive exactly one assignment.

---

# 2. Compact Constructor

One of the most elegant features of Records is the **Compact Constructor**.

Instead of repeating assignments,

Java allows you to write only the validation logic.

Example:

```java
public record Employee(

        Long id,
        String name

) {

    public Employee {

        if (id == null) {

            throw new IllegalArgumentException(
                    "Id cannot be null"
            );

        }

    }

}
```

Notice something unusual.

There are:

- No parameters
- No assignments

Yet this compiles successfully.

---

# What Does the Compiler Do?

Internally,

the compiler expands the compact constructor into:

```java
public Employee(

        Long id,
        String name

) {

    if (id == null) {

        throw new IllegalArgumentException(
                "Id cannot be null"
        );

    }

    this.id = id;
    this.name = name;

}
```

The assignments are inserted automatically.

---

# Why Was the Compact Constructor Introduced?

Validation is extremely common.

Without compact constructors:

```java
public Employee(

        Long id,
        String name

) {

    if (id == null) {

        throw new IllegalArgumentException();

    }

    this.id = id;
    this.name = name;

}
```

Notice how the assignments are repetitive.

Compact constructors eliminate this boilerplate.

---

# Constructor Execution Order

Suppose we have:

```java
public record Employee(

        Long id,
        String name

) {

    public Employee {

        System.out.println("Validation");

    }

}
```

Creating:

```java
new Employee(
        1L,
        "Guru"
);
```

Execution order:

```
Memory Allocation

        │

        ▼

Compact Constructor Executes

        │

        ▼

Compiler Inserts Assignments

        │

        ▼

Object Returned
```

Notice something important.

Your validation executes **before** the compiler-generated assignments.

---

# Can We Modify Parameters?

Yes.

One of the most useful features of compact constructors is that you may modify the parameters before they are assigned.

Example:

```java
public record Employee(

        String name

) {

    public Employee {

        name = name.trim();

    }

}
```

Compiler expansion:

```java
public Employee(

        String name

) {

    name = name.trim();

    this.name = name;

}
```

Output:

```java
Employee employee =
        new Employee("  Guru  ");

System.out.println(employee.name());
```

```
Guru
```

---

# Another Example

Normalize email addresses.

```java
public record User(

        String email

) {

    public User {

        email = email.toLowerCase();

    }

}
```

Now every object automatically stores lowercase email addresses.

---

# Validation Example

Age validation.

```java
public record Employee(

        int age

) {

    public Employee {

        if (age < 18) {

            throw new IllegalArgumentException(
                    "Invalid age"
            );

        }

    }

}
```

This guarantees that invalid objects can never be created.

---

# 3. Custom (Overloaded) Constructors

Records may also declare additional constructors.

Example:

```java
public record Employee(

        Long id,
        String name

) {

    public Employee(String name) {

        this(0L, name);

    }

}
```

Usage:

```java
Employee employee =
        new Employee("Guru");
```

Internally,

the overloaded constructor delegates to the canonical constructor.

---

# Important Rule

Every overloaded constructor **must eventually call** the canonical constructor.

This is illegal:

```java
public Employee(String name) {

}
```

Compilation Error.

Because every Record Component must be initialized.

---

# Constructor Chaining

Example:

```java
public record Employee(

        Long id,
        String name

) {

    public Employee(String name) {

        this(100L, name);

    }

}
```

Execution:

```
Employee(String)

        │

        ▼

Canonical Constructor

        │

        ▼

Initialize Components
```

This ensures all initialization logic remains centralized.

---

# Defensive Copying

One of the most important enterprise topics.

Consider:

```java
public record Employee(

        List<String> skills

) {}
```

Now:

```java
List<String> list =
        new ArrayList<>();

list.add("Java");

Employee employee =
        new Employee(list);
```

Later:

```java
list.add("Spring Boot");
```

What happens?

```java
employee.skills()
```

now contains:

```
Java

Spring Boot
```

Why?

Because the field is final,

but the **List object itself remains mutable**.

This is called **shallow immutability**.

---

# Defensive Copying Using Compact Constructor

Solution:

```java
public record Employee(

        List<String> skills

) {

    public Employee {

        skills = List.copyOf(skills);

    }

}
```

Now:

```java
Employee employee =
        new Employee(list);
```

Any later modifications to the original list do not affect the Record.

---

# Why Is This Important?

Many Spring Boot applications expose DTOs through REST APIs.

Without defensive copying,

outside code may unintentionally modify internal state.

Immutable DTOs improve:

- Thread safety
- Predictability
- Reliability
- Security

---

# Common Constructor Mistakes

## Mistake 1

Trying to assign fields manually inside a compact constructor.

Incorrect:

```java
public Employee {

    this.name = name;

}
```

Compilation Error.

The compiler performs the assignments automatically.

---

## Mistake 2

Forgetting to delegate from overloaded constructors.

Incorrect:

```java
public Employee(String name) {

}
```

Compilation Error.

Always call:

```java
this(...);
```

---

## Mistake 3

Ignoring mutable collections.

```java
record Employee(

        List<String> skills

) {}
```

This Record is only **shallowly immutable**.

Always create defensive copies when exposing mutable data structures.

---

# Key Takeaways

- Every Record automatically receives a **Canonical Constructor**.
- Developers may replace the canonical constructor.
- Compact constructors are intended for validation and normalization.
- The compiler automatically inserts field assignments in compact constructors.
- Overloaded constructors must delegate to the canonical constructor.
- Compact constructors are the preferred way to validate Record components.
- Records are **shallowly immutable** by default.
- Use defensive copying for mutable collections to achieve deep immutability.

# Immutability in Records

One of the biggest misconceptions about Records is the statement:

> **"Records are immutable."**

This statement is **partially correct**.

A more accurate statement is:

> **Records are shallowly immutable by default.**

Understanding the difference between **shallow** and **deep** immutability is extremely important in enterprise applications and Java interviews.

---

# What Is Immutability?

An object is **immutable** if its state cannot change after it has been created.

Example:

```java
String name = "Guru";
```

When you execute:

```java
name = "Alex";
```

You are **not modifying** the original `String`.

Instead,

a new `String` object is created, and `name` now refers to that new object.

This makes `String` immutable.

---

# Why Is Immutability Important?

Immutable objects offer several advantages.

- Thread-safe
- Easier to reason about
- No synchronization required
- Predictable behavior
- Safe to share across threads
- Easier caching
- Easier testing

This is one reason why Records are heavily used for DTOs.

---

# Shallow Immutability

A Record guarantees that **its fields cannot be reassigned**.

Consider:

```java
public record Employee(

        String name,
        int age

) {}
```

Usage:

```java
Employee employee =
        new Employee(
                "Guru",
                25
        );
```

Trying to modify a component:

```java
employee.name = "Alex";
```

Compilation Error.

The field is `private final`.

So the reference cannot change.

---

# But What About Mutable Objects?

Now consider:

```java
public record Employee(

        List<String> skills

) {}
```

Usage:

```java
List<String> skills =
        new ArrayList<>();

skills.add("Java");

Employee employee =
        new Employee(skills);
```

Later:

```java
skills.add("Spring Boot");
```

Now inspect the Record.

```java
System.out.println(employee.skills());
```

Output:

```
[Java, Spring Boot]
```

The Record changed!

---

# Why Did This Happen?

Let's examine the memory layout.

```
                Stack

skills  ───────────────┐

employee ───────┐       │
                │       │
                ▼       ▼

               Heap

         Employee Record

skills ───────────────┐
                      │
                      ▼

              ArrayList Object

            [Java, Spring Boot]
```

Both variables point to the **same ArrayList object**.

The Record protects the **reference**, not the object itself.

---

# Understanding the Difference

The following is illegal:

```java
employee.skills = new ArrayList<>();
```

because the field is `final`.

However,

this is perfectly legal:

```java
employee.skills().add("Docker");
```

Why?

Because the `ArrayList` itself remains mutable.

This is exactly what **shallow immutability** means.

---

# Deep Immutability

An object is **deeply immutable** when:

- Its references cannot change.
- The objects referenced by those fields also cannot change.

For example:

```java
public record Employee(

        List<String> skills

) {

    public Employee {

        skills = List.copyOf(skills);

    }

}
```

Now:

```java
List<String> skills =
        new ArrayList<>();

skills.add("Java");

Employee employee =
        new Employee(skills);

skills.add("Spring Boot");
```

Output:

```java
employee.skills()
```

```
[Java]
```

The Record remains unchanged.

---

# Why Does `List.copyOf()` Help?

`List.copyOf()` creates an **unmodifiable copy** of the supplied collection.

Instead of storing:

```
Original List
```

the Record stores:

```
New Immutable List
```

Memory becomes:

```
Original List

        │

        ▼

[Java, Spring Boot]


Employee Record

        │

        ▼

Immutable List

[Java]
```

Now modifications to the original list do not affect the Record.

---

# Defensive Copying

This technique is called **Defensive Copying**.

Instead of trusting external objects,

the Record creates its own safe copy.

This is considered a best practice whenever Record Components contain:

- List
- Set
- Map
- Arrays
- Mutable custom objects

---

# Mutable Custom Objects

Consider:

```java
class Address {

    private String city;

    // getter & setter

}
```

Now:

```java
public record Employee(

        Address address

) {}
```

Usage:

```java
Address address =
        new Address();

address.setCity("Chennai");

Employee employee =
        new Employee(address);
```

Later:

```java
address.setCity("Bangalore");
```

The Record now exposes:

```
Bangalore
```

Again,

only the reference is protected.

The `Address` object itself remains mutable.

---

# Achieving Deep Immutability

Possible approaches include:

- Use immutable objects (`String`, `LocalDate`, `UUID`, etc.)
- Use `List.copyOf()`
- Use immutable collections
- Design custom immutable classes
- Perform defensive copying in constructors

---

# Records and JPA

A common interview question is:

> **Can Records be used as JPA entities?**

The answer is generally:

**No.**

---

# Why?

JPA requires:

- A no-argument constructor
- Mutable fields
- Lazy-loading proxies
- Setter methods (or field mutation)

Records violate several of these assumptions.

For example,

this entity is invalid:

```java
@Entity
public record Employee(

        @Id
        Long id,

        String name

) {}
```

Hibernate cannot manage it as a typical entity.

---

# Where Should Records Be Used?

Excellent choices:

- Request DTOs
- Response DTOs
- Kafka events
- RabbitMQ messages
- REST payloads
- Configuration properties
- Projection interfaces
- Value objects

Poor choices:

- JPA entities
- Stateful domain models
- Objects requiring mutation
- ORM-managed entities

---

# Performance

Do Records improve runtime performance?

No.

A Record is still a normal Java object.

Memory layout is essentially identical to a regular class containing the same fields.

The advantages come from:

- Less source code
- Fewer bugs
- Better readability
- Easier maintenance

Not from faster execution.

---

# Best Practices

- Use Records for immutable data carriers.
- Prefer immutable component types (`String`, `UUID`, `LocalDate`, etc.).
- Perform defensive copying for mutable collections.
- Use compact constructors for validation and normalization.
- Keep business logic minimal; Records should primarily model data.
- Use Records for API contracts and messaging payloads.
- Avoid using Records as JPA entities.

---

# Common Mistakes

## Mistake 1

Assuming Records are deeply immutable.

They are only **shallowly immutable** by default.

---

## Mistake 2

Using mutable collections without defensive copying.

Always consider:

```java
List.copyOf()
```

---

## Mistake 3

Adding business logic unrelated to the data model.

Records are intended to represent state, not rich domain behavior.

---

## Mistake 4

Trying to use Records for ORM entities.

Use traditional classes for JPA/Hibernate entities.

---

## Key Takeaways

- Records are **shallowly immutable**.
- Use defensive copying to achieve deep immutability.
- Records integrate seamlessly with Jackson and Spring Boot.
- Bean Validation works directly on Record Components.
- Records are excellent DTOs but poor JPA entities.
- Serialization works naturally because Records fully define their state.
- The main benefit of Records is improved maintainability, not runtime performance.

# Interview Questions and Answers

The following questions are commonly asked in Java 17/21 backend interviews. They range from basic understanding to senior-level design discussions.

---

# Q1. What is a Record?

### Answer

A Record is a **special type of class** introduced in Java 16 (JEP 395) that is designed to model immutable data.

Instead of writing boilerplate code for:

- fields
- constructor
- accessors
- `equals()`
- `hashCode()`
- `toString()`

the compiler generates these members automatically.

Example:

```java
public record Employee(

        Long id,
        String name

) {}
```

---

# Q2. Is a Record a class?

### Answer

Yes.

A Record is a real Java class.

Internally it behaves similarly to:

```java
public final class Employee
        extends Record {

}
```

The compiler simply generates most of the implementation.

---

# Q3. Does a Record extend `Object`?

### Answer

Indirectly.

Every Record extends:

```
java.lang.Record
```

and `Record` itself extends:

```
java.lang.Object
```

Hierarchy:

```
Object
   ▲
Record
   ▲
Employee
```

---

# Q4. Why was `java.lang.Record` introduced?

### Answer

It provides a common superclass for all Records.

This allows:

- Reflection support
- JVM identification
- Framework integration
- Consistent language behavior

---

# Q5. Can a Record extend another class?

### Answer

No.

Java supports only single inheritance.

Since every Record already extends:

```java
java.lang.Record
```

it cannot extend any other class.

---

# Q6. Can another class extend a Record?

### Answer

No.

Every Record is implicitly `final`.

Final classes cannot be subclassed.

---

# Q7. Can Records implement interfaces?

### Answer

Yes.

Example:

```java
interface Printable {

    void print();

}
```

```java
public record Employee(

        String name

) implements Printable {

    @Override
    public void print() {

        System.out.println(name);

    }

}
```

This is one of the preferred ways to achieve polymorphism with Records.

---

# Q8. Why are Records final?

### Answer

To preserve:

- immutability
- value-based equality
- predictable generated methods

Allowing inheritance would make generated methods like `equals()` ambiguous.

---

# Q9. Can Records have methods?

### Answer

Yes.

They may define:

- instance methods
- static methods
- private methods
- overridden accessors

Example:

```java
public boolean isAdult() {

    return age >= 18;

}
```

---

# Q10. Can Records have constructors?

### Answer

Yes.

Three types are supported:

- Canonical constructor
- Compact constructor
- Overloaded constructor

---

# Q11. What is a Canonical Constructor?

### Answer

A constructor whose parameters exactly match the Record Components.

Example:

```java
record User(

        String name,
        int age

) {}
```

Canonical constructor:

```java
User(String name, int age)
```

---

# Q12. What is a Compact Constructor?

### Answer

A constructor without an explicit parameter list that is primarily used for validation or normalization.

Example:

```java
public User {

    if (age < 18) {

        throw new IllegalArgumentException();

    }

}
```

The compiler automatically inserts the field assignments.

---

# Q13. Why use a Compact Constructor instead of a Canonical Constructor?

### Answer

It eliminates repetitive assignments.

Instead of writing:

```java
this.name = name;
```

for every component,

the compiler performs these assignments automatically.

The developer focuses only on validation or transformation.

---

# Q14. Can Record Components be modified inside a Compact Constructor?

### Answer

Yes.

Example:

```java
public User {

    name = name.trim();

}
```

The compiler assigns the modified value to the field.

---

# Q15. What methods does the compiler generate?

### Answer

For every Record the compiler generates:

- private final fields
- canonical constructor
- accessor methods
- `equals()`
- `hashCode()`
- `toString()`

---

# Q16. Why do Records have accessors instead of getters?

### Answer

Records are not based on JavaBean conventions.

Instead of:

```java
getName()
```

they expose:

```java
name()
```

This makes the API more concise and aligns with the Record Component names.

---

# Q17. Are Records immutable?

### Answer

They are **shallowly immutable**.

The references cannot change,

but the referenced objects may still be mutable.

---

# Q18. Explain shallow vs deep immutability.

### Answer

Shallow immutability:

```java
record Employee(List<String> skills) {}
```

The `skills` reference cannot change, but the `List` can.

Deep immutability:

Create defensive copies using:

```java
List.copyOf(skills)
```

or use immutable component types.

---

# Q19. Why is defensive copying important?

### Answer

Without defensive copying, external code can modify mutable objects after a Record has been created.

This breaks the expectation of immutability and can introduce subtle bugs.

---

# Q20. Can Records be used with Spring Boot?

### Answer

Yes.

Excellent use cases include:

- Request DTOs
- Response DTOs
- Configuration Properties
- Kafka messages
- RabbitMQ messages
- REST payloads

---

# Q21. Can Records be used as JPA entities?

### Answer

Generally no.

Reasons include:

- no no-arg constructor
- immutable state
- proxy generation issues
- JPA expects mutable entities

Traditional classes are the recommended approach for entities.

---

# Q22. Do Records improve runtime performance?

### Answer

No.

The generated bytecode is similar to an equivalent manually written class.

The advantages are:

- reduced boilerplate
- fewer bugs
- easier maintenance

---

# Q23. How do frameworks know a class is a Record?

### Answer

Using Reflection.

Example:

```java
Employee.class.isRecord();
```

Output:

```
true
```

Frameworks can also inspect components using:

```java
getRecordComponents()
```

---

# Q24. Can Records contain static members?

### Answer

Yes.

They may define:

- static fields
- static methods
- nested types

Only additional **instance fields** are prohibited.

---

# Q25. Can Records contain instance fields?

### Answer

No.

All instance state must be declared as Record Components.

Otherwise the generated methods would become inconsistent.

---

# Q26. Can you override `equals()` or `toString()`?

### Answer

Yes.

Although the compiler generates these methods,

developers may override them when necessary.

However, doing so should be rare because the generated implementations are usually correct.

---

# Q27. Can Records participate in polymorphism?

### Answer

Yes.

Not through class inheritance,

but through interface implementation.

Example:

```java
Shape shape =
        new Circle(10);
```

where `Circle` is a Record implementing `Shape`.

---

# Q28. How are Records different from Lombok's `@Data`?

### Answer

| Record | Lombok `@Data` |
|---------|----------------|
| Java language feature | Annotation processor |
| Immutable by default | Mutable by default |
| Extends `Record` | Normal class |
| Built into JDK | Requires Lombok dependency |
| Compiler-generated | Source code generated before compilation |

---

# Q29. When should you choose a Record over a class?

### Answer

Use a Record when:

- the object represents immutable data
- identity is based on values
- behavior is minimal
- you want concise code

Use a class when:

- mutable state is required
- inheritance is required
- JPA manages the object
- complex business behavior exists

---

# Q30. Summarize Records in one sentence.

### Answer

> **A Record is a special immutable data carrier whose state is declared once and whose boilerplate implementation is generated automatically by the Java compiler.**

---

# Advanced Interview Scenarios

## Scenario 1

### Question

Why didn't Java simply improve Lombok instead of introducing Records?

### Answer

Lombok is a third-party annotation processor.

Records are:

- part of the Java Language Specification
- supported by the compiler
- supported by the JVM ecosystem
- understood by Reflection APIs
- supported by IDEs without plugins

Records provide a standardized language feature rather than relying on external tooling.

---

## Scenario 2

### Question

If Records are immutable, why can a `List` inside a Record still change?

### Answer

Because the Record only protects the **reference**.

The referenced `List` object remains mutable unless a defensive copy or immutable collection is used.

This distinction is the difference between shallow and deep immutability.

---

## Scenario 3

### Question

Can you use Records in a microservices architecture?

### Answer

Yes, and they are often an excellent choice.

Typical uses include:

- REST request/response models
- Kafka events
- RabbitMQ messages
- gRPC payloads
- configuration objects

Their immutability and concise syntax make them ideal for data transfer across service boundaries.

# Chapter Summary

Records, introduced in **Java 16**, provide a concise and standardized way to model immutable data. They eliminate repetitive boilerplate by allowing the compiler to generate fields, constructors, accessor methods, `equals()`, `hashCode()`, and `toString()`. Records are ordinary Java classes that implicitly extend `java.lang.Record` and are implicitly `final`, making them ideal for value-based objects such as DTOs, API payloads, configuration classes, and messaging events.

While Records offer shallow immutability by default, developers must still use defensive copying for mutable components to achieve deep immutability. They integrate seamlessly with Spring Boot, Jackson, Bean Validation, and modern Java tooling, but they are generally unsuitable as JPA entities due to ORM requirements for mutable state and proxy generation.

Understanding Records involves much more than their syntax—it requires knowledge of compiler-generated members, constructor variants, inheritance rules, reflection support, and practical backend usage. Mastering these concepts prepares you for modern Java 17/21 backend development and technical interviews.

# Introduction

Inheritance is one of the fundamental pillars of Object-Oriented Programming (OOP).

One of Java's biggest strengths is that developers can create extensible class hierarchies.

For example,

```java
class Animal {

}
```

can be extended as:

```java
class Dog extends Animal {

}
```

or

```java
class Cat extends Animal {

}
```

or

```java
class Bird extends Animal {

}
```

This flexibility has been one of Java's greatest advantages for decades.

However,

there are many situations where unrestricted inheritance becomes a disadvantage rather than a benefit.

Consider a payment processing system.

```
           Payment

               ▲

      ┌────────┼────────┐

      ▼        ▼        ▼

 CreditCard  UPI     NetBanking
```

Suppose your application supports **only three payment methods**.

Should another developer be allowed to create:

```java
class BitcoinPayment extends Payment {

}
```

or

```java
class CryptoPayment extends Payment {

}
```

or

```java
class CashPayment extends Payment {

}
```

Probably not.

The hierarchy should remain closed because the application logic only understands the officially supported payment types.

This problem existed in Java for many years.

To solve it,

Java introduced **Sealed Classes**.

---

# Why Were Sealed Classes Introduced?

Before Java 17,

Java offered only two inheritance choices.

### Option 1 – Open Inheritance

```java
class Animal {

}
```

Anyone could extend it.

```
Animal

   ▲

Anyone can create subclasses
```

Advantages:

- Highly extensible
- Flexible
- Framework friendly

Disadvantages:

- No control over hierarchy
- Difficult to reason about
- Exhaustiveness impossible
- API can be misused

---

### Option 2 – Final Classes

```java
final class Animal {

}
```

Now nobody can extend it.

Advantages:

- Complete control
- Immutable hierarchy

Disadvantages:

- No extensibility at all

---

Java lacked a middle ground.

Developers wanted to say:

> "Only **these specific classes** may extend this class."

That is exactly what Sealed Classes provide.

---

# The Problem Before Sealed Classes

Imagine a banking application.

```java
abstract class Account {

}
```

Your application officially supports:

- Savings Account
- Current Account
- Fixed Deposit

Ideally,

the hierarchy should be:

```
             Account

      ┌────────┼────────┐

      ▼        ▼        ▼

 Savings   Current   FixedDeposit
```

But Java previously allowed anyone to write:

```java
class CryptoAccount extends Account {

}
```

or

```java
class LotteryAccount extends Account {

}
```

Your application now has subclasses it was never designed to handle.

---

# Real Enterprise Example

Consider an authentication system.

```
AuthenticationProvider

        ▲

 ┌──────┼────────┐

 ▼      ▼        ▼

OAuth  LDAP   Database
```

If another developer introduces:

```java
class SMSAuthentication
        extends AuthenticationProvider {

}
```

Your authentication flow may fail because it does not recognize this provider.

The hierarchy should be restricted.

---

# Design Goals

Sealed Classes were introduced with several objectives.

---

## Goal 1 – Restrict Inheritance

Instead of allowing everyone to extend a class,

allow only explicitly approved subclasses.

---

## Goal 2 – Model Finite Hierarchies

Some domains naturally contain a fixed number of subtypes.

Examples include:

- Payment Methods
- Authentication Providers
- Network Protocols
- Expression Trees
- Compiler AST Nodes
- Transport Types

These are perfect candidates for Sealed Classes.

---

## Goal 3 – Improve Pattern Matching

Modern Java introduces Pattern Matching.

Suppose:

```java
switch (payment) {

}
```

If the compiler knows all possible subclasses,

it can verify whether every case has been handled.

This is called **Exhaustiveness Checking**.

We'll study this in detail in the Pattern Matching chapter.

---

## Goal 4 – Better API Design

Library authors can expose extension points while still protecting the integrity of their APIs.

Instead of unlimited inheritance,

they explicitly document the supported hierarchy.

---

# Non-Goals

Sealed Classes were **not** introduced to replace:

- Interfaces
- Abstract classes
- Final classes

Instead,

they complement these features.

Each still has its own purpose.

---

# What Is a Sealed Class?

A Sealed Class is a class that explicitly controls **which other classes are allowed to extend it**.

Example:

```java
public sealed class Payment

        permits CreditCard,
                UPI,
                NetBanking {

}
```

Here,

only the following classes may extend `Payment`:

- `CreditCard`
- `UPI`
- `NetBanking`

Any other subclass results in a compilation error.

---

# Understanding the `permits` Clause

The `permits` clause is the heart of Sealed Classes.

Example:

```java
public sealed class Vehicle

        permits Car,
                Bike,
                Truck {

}
```

The compiler records that only these three subclasses are valid.

Attempting:

```java
class Bus extends Vehicle {

}
```

results in:

```
Compilation Error
```

because `Bus` is not listed in the `permits` clause.

---

# Architecture

Without Sealed Classes:

```
                Animal

                   ▲

      Anyone can create subclasses
```

With Sealed Classes:

```
                Payment

                   ▲

      ┌────────────┼────────────┐

      ▼            ▼            ▼

 CreditCard       UPI      NetBanking

Only these classes are permitted.
```

Inheritance becomes controlled rather than completely open.

# Why Not Just Use an Enum?

A common interview question.

Consider:

```java
enum PaymentType {

    CARD,
    UPI,
    NETBANKING

}
```

Enums only represent **constants**.

They cannot easily model rich behavior.

A Sealed hierarchy allows each subtype to implement its own logic.

Example:

```java
sealed interface Payment

        permits Card,
                UPI {

    void process();

}
```

Each implementation can contain different algorithms while still restricting the hierarchy.

| Final Class | Sealed Class |
|-------------|--------------|
| No subclasses allowed | Only approved subclasses allowed |
| Completely closed | Partially open |
| Maximum restriction | Controlled extensibility |
| Suitable for utility/value classes | Suitable for finite hierarchies |

| Abstract Class | Sealed Class |
|----------------|--------------|
| Anyone may extend | Only permitted subclasses may extend |
| Defines shared behavior | Controls hierarchy |
| Focus on abstraction | Focus on inheritance restriction |
| Often combined with `sealed` | Can itself be abstract |

# Key Points

- Introduced in **Java 17** (JEP 409).
- Restricts inheritance to approved subclasses.
- Uses the `sealed` keyword.
- Uses the `permits` clause to declare allowed subclasses.
- Ideal for modeling finite hierarchies.
- Improves API safety and maintainability.
- Enables future compiler features like exhaustive pattern matching.
- Does **not** replace abstract classes or interfaces.

# The Three New Modifiers

Sealed Classes introduced **three new language modifiers**.

They are:

- `sealed`
- `non-sealed`
- `final`

These three modifiers work together to define **how inheritance continues** after a sealed class.

Think of them as inheritance control keywords.

```
                 sealed

                    │

         ┌──────────┼──────────┐

         ▼          ▼          ▼

      sealed     final    non-sealed
```

Every permitted subclass **must choose exactly one** of these modifiers.

This is one of the most frequently asked interview topics.

---

# 1. The `sealed` Modifier

The `sealed` modifier restricts which classes are allowed to inherit.

Example:

```java
public sealed class Payment

        permits CardPayment,
                UpiPayment,
                NetBankingPayment {

}
```

Here,

only three subclasses are allowed.

```
             Payment

      ┌────────┼────────┐

      ▼        ▼        ▼

    Card      UPI    NetBanking
```

Attempting to create:

```java
class CryptoPayment
        extends Payment {

}
```

results in a compilation error because it is not listed in the `permits` clause.

---

# Why Is `sealed` Needed?

Before Java 17,

inheritance was either:

```
Open

or

Closed
```

There was no middle ground.

`sealed` provides **controlled inheritance**.

---

# 2. The `final` Modifier

A permitted subclass may choose to terminate the hierarchy.

Example:

```java
public final class CardPayment

        extends Payment {

}
```

Now,

no further subclasses are allowed.

```
Payment

   ▲

CardPayment

   ✖

No further inheritance
```

This completely closes that branch.

---

# Why Use `final`?

Suppose:

```java
CardPayment
```

already represents the complete implementation.

There is no valid subtype such as:

```
VisaCardPayment

MasterCardPayment

RuPayCardPayment
```

The hierarchy ends here.

---

# 3. The `non-sealed` Modifier

Sometimes,

one branch should remain extensible.

Example:

```java
public non-sealed class CardPayment

        extends Payment {

}
```

Now,

inheritance becomes open again.

```
Payment

    ▲

CardPayment

    ▲

Anyone may extend
```

Developers may now create:

```java
class VisaCard
        extends CardPayment {

}
```

or

```java
class MasterCard
        extends CardPayment {

}
```

---

# Why Not Simply Omit the Modifier?

A common interview question.

Suppose we write:

```java
class CardPayment
        extends Payment {

}
```

Compilation Error.

Every permitted subclass **must explicitly declare** one of:

- `final`
- `sealed`
- `non-sealed`

The compiler forces the developer to decide the future of the hierarchy.

---

# Why Is This Required?

Imagine the compiler allowed:

```java
class CardPayment
        extends Payment {

}
```

Should inheritance continue?

Should it stop?

The compiler cannot guess the developer's intention.

Therefore,

Java requires an explicit decision.


# The `permits` Clause

The `permits` clause specifies every permitted subclass.

Example:

```java
public sealed class Animal

        permits Dog,
                Cat,
                Bird {

}
```

The compiler records this list.

Only these classes may inherit.

---

# Can We Omit `permits`?

Yes,

if every permitted subclass is declared in the same source file.

Example:

```java
public sealed class Animal {

}

final class Dog
        extends Animal {

}

final class Cat
        extends Animal {

}
```

The compiler automatically discovers the subclasses.

This is called **implicit permits**.

---

# Package Rule

Another important interview topic.

When not using the same source file,

every permitted subclass must be located in:

- the same package (unnamed module), or
- the same named module.

Example:

```
com.company.payment

Payment

CardPayment

UPIPayment
```

This compiles.

However,

placing a permitted subclass in an unrelated package without the appropriate module relationship is not allowed.

The compiler must be able to verify the hierarchy during compilation.

---

# Why This Restriction Exists

The compiler validates the entire sealed hierarchy.

If subclasses could exist anywhere,

the compiler could never determine whether the hierarchy was complete.

Restricting the package/module makes the hierarchy fully known.

---

# JVM Internals

An important interview question is:

> **Does the JVM understand sealed classes?**

Answer:

Yes.

Unlike `var`, which is purely a compiler feature,

Sealed Classes are represented in the JVM.

Java 17 introduced a new class-file attribute:

```
PermittedSubclasses
```

This attribute stores the list of permitted subclasses.

Example (conceptually):

```
Payment.class

PermittedSubclasses

CardPayment

UpiPayment

NetBankingPayment
```

The JVM can then enforce the inheritance restrictions at class-loading time.

Java Reflection provides APIs specifically for sealed classes.

Example:

```java
Class<?> clazz =
        Payment.class;
```

Check if it is sealed:

```java
System.out.println(
        clazz.isSealed()
);
```

Output:

```
true
```

Reflection also allows inspection of the hierarchy.

```java
Class<?>[] subclasses =
        Payment.class
               .getPermittedSubclasses();
```

Example output:


# Key Takeaways

- Sealed Classes introduce three modifiers:
    - `sealed`
    - `final`
    - `non-sealed`
- Every permitted subclass must explicitly choose one of these modifiers.
- `sealed` continues a restricted hierarchy.
- `final` terminates a hierarchy branch.
- `non-sealed` reopens inheritance completely.
- The `permits` clause explicitly defines allowed subclasses.
- The compiler validates the hierarchy during compilation.
- The JVM stores permitted subclasses using the `PermittedSubclasses` class-file attribute.
- Reflection APIs provide `isSealed()` and `getPermittedSubclasses()` for runtime inspection.

# Pattern Matching and Sealed Classes

One of the primary motivations for introducing Sealed Classes was to improve **Pattern Matching**.

Imagine a payment hierarchy.

```java

        permits CardPayment,
                UpiPayment,
                NetBankingPayment {

```

```
     ▲

```

```
      ▲

 ┌────┼──────────┐
```

# What Is Exhaustiveness Checking?

Exhaustiveness checking means:

> The compiler verifies that every possible subtype has been handled.

Example:

```java
switch (payment) {

}
```

Since the compiler knows all subclasses,

it can determine whether any case is missing.

---

# Example

```java

        permits Circle,
                Rectangle,
                Triangle {
```

```java
        case Rectangle r -> ...

```

The compiler verifies that:

- Circle ✔
- Rectangle ✔
- Triangle ✔

Every subtype has been covered.

# Interview Questions and Answers

The following questions are among the most frequently asked in Java 17/21 backend interviews. They range from fundamental concepts to senior-level design discussions.

---

# Q1. What is a Sealed Class?

### Answer

A Sealed Class is a class that **restricts which other classes or interfaces are allowed to inherit from or implement it**.

Instead of allowing unrestricted inheritance, the developer explicitly declares the permitted subclasses using the `permits` clause.

Example:

```java
public sealed class Payment

        permits CardPayment,
                UpiPayment,
                NetBankingPayment {

}
```

---

# Q2. Why were Sealed Classes introduced?

### Answer

Before Java 17, developers had only two choices:

- Open inheritance
- No inheritance (`final`)

There was no way to allow **only a limited set of subclasses**.

Sealed Classes fill this gap by providing **controlled inheritance**.

---

# Q3. Which Java version introduced Sealed Classes?

### Answer

- Preview: Java 15
- Second Preview: Java 16
- Final Release: Java 17

JEP:

```
409
```

---

# Q4. What problem do Sealed Classes solve?

### Answer

They solve the problem of uncontrolled inheritance.

For example,

a payment system may support only:

- Card
- UPI
- NetBanking

Sealed Classes prevent developers from introducing unsupported subclasses such as:

```java
CryptoPayment
```

or

```java
CashPayment
```

---

# Q5. What is the `permits` clause?

### Answer

The `permits` clause explicitly lists every class that is allowed to extend the sealed class.

Example:

```java
sealed class Animal

        permits Dog,
                Cat {

}
```

Only `Dog` and `Cat` may extend `Animal`.

---

# Q6. Can the `permits` clause be omitted?

### Answer

Yes.

If all permitted subclasses are declared in the **same source file**, the compiler can infer them automatically.

Otherwise, the `permits` clause is required.

---

# Q7. What modifiers can a permitted subclass use?

### Answer

Every permitted subclass **must** explicitly declare one of:

- `final`
- `sealed`
- `non-sealed`

Example:

```java
final class Dog extends Animal {}
```

or

```java
sealed class Dog

        extends Animal

        permits Puppy {

}
```

or

```java
non-sealed class Dog

        extends Animal {

}
```

---

# Q8. Why must every permitted subclass choose one of these modifiers?

### Answer

The compiler requires developers to explicitly decide the future of each inheritance branch.

Without this rule, it would be unclear whether inheritance should continue or stop.

---

# Q9. What does `non-sealed` mean?

### Answer

`non-sealed` reopens inheritance.

Example:

```java
non-sealed class CardPayment

        extends Payment {

}
```

Now any class may extend `CardPayment`.

---

# Q10. What is the difference between `sealed` and `final`?

### Answer

`sealed`

- Allows only approved subclasses.

`final`

- Allows no subclasses.

---

# Q11. Can interfaces be sealed?

### Answer

Yes.

Example:

```java
sealed interface Payment

        permits CardPayment,
                UpiPayment {

}
```

Only the listed implementations are allowed.

---

# Q12. Why are sealed interfaces commonly preferred?

### Answer

Modern Java encourages programming to interfaces.

A sealed interface provides:

- loose coupling
- polymorphism
- controlled implementations

This combination is ideal for enterprise applications.

---

# Q13. Can abstract classes be sealed?

### Answer

Yes.

Example:

```java
public abstract sealed class Shape

        permits Circle,
                Rectangle {

}
```

The class remains abstract while still controlling its hierarchy.

---

# Q14. Can a Sealed Class be instantiated?

### Answer

Yes, if it is a concrete class.

If it is also `abstract`, then it cannot be instantiated.

Sealing and abstraction are independent concepts.

---

# Q15. Can a Sealed Class extend another class?

### Answer

Yes.

Example:

```java
class Vehicle {

}

sealed class Car

        extends Vehicle

        permits Sedan {

}
```

This is perfectly valid.

---

# Q16. Can a Sealed Class implement interfaces?

### Answer

Yes.

Example:

```java
sealed class CardPayment

        implements Payment

        permits VisaCard {

}
```

---

# Q17. Can a Record be a permitted subclass?

### Answer

Yes.

Since Records are implicitly `final`, they naturally fit sealed hierarchies.

Example:

```java
sealed interface Shape

        permits Circle {

}

record Circle(

        double radius

) implements Shape {

}
```

---

# Q18. Can enums participate in sealed hierarchies?

### Answer

Enums cannot extend classes because they already extend `java.lang.Enum`.

However, enums can implement sealed interfaces if they are explicitly permitted.

Example:

```java
sealed interface Payment

        permits PaymentType {

}

enum PaymentType implements Payment {

    CARD,
    UPI

}
```

---

# Q19. How do Sealed Classes improve Pattern Matching?

### Answer

The compiler knows every permitted subtype.

This enables **exhaustiveness checking**.

Every possible subtype must be handled in a `switch`.

---

# Q20. What is exhaustiveness checking?

### Answer

The compiler verifies that every permitted subtype has been handled.

Example:

```java
switch(shape) {

    case Circle c -> ...

    case Rectangle r -> ...

}
```

If another subtype exists,

the compiler reports an error.

---

# Q21. Do Sealed Classes improve runtime performance?

### Answer

No.

Their benefits are:

- compile-time safety
- maintainability
- API design
- better pattern matching

Object execution speed is essentially unchanged.

---

# Q22. Do Sealed Classes affect memory layout?

### Answer

No.

Objects have the same memory layout as ordinary classes.

Only the class metadata changes.

---

# Q23. How does the JVM know a class is sealed?

### Answer

The compiler writes a special class-file attribute:

```
PermittedSubclasses
```

The JVM reads this metadata during class loading.

---

# Q24. What Reflection APIs support Sealed Classes?

### Answer

Two important methods are:

```java
Class::isSealed()
```

and

```java
Class::getPermittedSubclasses()
```

---

# Q25. Can Sealed Classes be serialized?

### Answer

Yes.

Serialization works exactly like ordinary classes.

The sealing restriction does not interfere with serialization.

---

# Q26. Can Jackson serialize Sealed Classes?

### Answer

Yes.

However,

polymorphic serialization usually requires type metadata such as:

```java
@JsonTypeInfo
```

so that Jackson knows which subtype to instantiate during deserialization.

---

# Q27. Can Hibernate entities be sealed?

### Answer

Generally,

no.

Hibernate often generates runtime proxy subclasses.

Those proxies are not listed in the `permits` clause,

which conflicts with the sealed hierarchy.

---

# Q28. What are the best use cases for Sealed Classes?

### Answer

Excellent choices include:

- Payment methods
- Authentication providers
- Workflow states
- Domain events
- AST nodes
- Commands
- Notifications

---

# Q29. When should you avoid Sealed Classes?

### Answer

Avoid them when:

- third parties are expected to add implementations
- frameworks rely on proxy subclasses
- unrestricted extensibility is a design goal

---

# Q30. How are Sealed Classes different from enums?

### Answer

Enums model **fixed constants**.

Sealed Classes model **fixed types with behavior**.

Each subtype can implement different algorithms while remaining part of a finite hierarchy.

# Chapter Summary

Sealed Classes, introduced in **Java 17**, provide a way to explicitly control inheritance by restricting which classes or interfaces may extend or implement a type. They fill the long-standing gap between completely open inheritance and completely closed (`final`) classes.

By combining the `sealed` keyword with the `permits` clause, developers can model finite business domains such as payment methods, authentication providers, workflow states, and compiler syntax trees. Each permitted subclass must explicitly decide whether to terminate inheritance (`final`), continue it in a controlled way (`sealed`), or reopen it (`non-sealed`).

Sealed Classes work especially well with Pattern Matching and Switch Expressions because the compiler can perform exhaustiveness checking, ensuring every valid subtype is handled at compile time. They are fully supported by the JVM through the `PermittedSubclasses` class-file attribute and expose dedicated reflection APIs for runtime inspection.

Although they do not improve runtime performance, Sealed Classes significantly improve API safety, maintainability, and correctness. They are a valuable tool in modern Java 17/21 backend development, particularly when designing finite, behavior-rich hierarchies.

The answer is:

**No.**

The difference is that the compiler generates the cast automatically.


```
    │

Execute Block
```

Now consider:

```java
Object object = "Guru";

if (object instanceof String value) {

    System.out.println(value.length());

}
```

Conceptually, the compiler transforms it into something similar to:

```java
if (object instanceof String) {

    String value = (String) object;

    System.out.println(value.length());

}
```

The JVM still performs the same operations.

The compiler simply writes the cast for you.

Pattern Matching is an example of **syntactic sugar**.

The compiler rewrites:


The JVM does **not** execute a special "pattern matching" instruction.

# Dominance Rules

One of the most common Java 21 interview questions.

Suppose:

```java
switch(value) {

    case Object o ->

            System.out.println("Object");

    case String s ->

            System.out.println("String");

}
```

Compilation Error.

---

# Why?

Every `String` is already an `Object`.

The first case matches everything.

The second case can never execute.

This is called:

```
Dominated Pattern
```

# Correct Ordering

Always place **more specific** patterns before **more general** patterns.

Correct:

```java

    case String s ->

            System.out.println("String");

    case Object o ->

            System.out.println("Object");

}
```

Execution:

```
String?

↓

Yes

↓

Done

Otherwise

↓

Object
```


```

↓

extends Number
```

Every Integer is already a Number.

The Integer case becomes unreachable.

```java

case Number n ->
```

Always order from **most specific** to **most general**.

# What is a Guarded Pattern?

A Guarded Pattern combines:

```
        +

Boolean Condition
```

```
      │

      │
```

```
    │
```

# Pattern Matching vs Polymorphism

This is one of the most important design interview topics.

Suppose:

```java

    void process();

}
```

Implementations:

```java

UpiPayment
```

Then:

```java

```

No Pattern Matching needed.

---

# Why?

Each subtype already knows how to process itself.

This is classic polymorphism.

---

# Pattern Matching Version

```java
switch(payment){

    case CardPayment c ->

            ...

    case UpiPayment u ->

            ...

}
```

Here,

behavior is centralized.

---

# Which Is Better?

Usually:

```
Polymorphism
```

Why?

Because:

- Open for extension
- Encapsulates behavior
- Follows the Open/Closed Principle
- Avoids type switching

---

# When Pattern Matching Is Better

Pattern Matching is preferable when:

- You cannot modify the hierarchy.
- Behavior belongs outside the domain objects.
- You're implementing reporting, logging, serialization, validation, or mapping.
- Different subsystems need different logic for the same types.

---

Domain:

```java

        permits Circle,
                Rectangle{

}
```

```

```

Adding:

```java

```

inside every domain object may not be desirable.

Instead:

```java

    case Rectangle r -> ...

}
```

The serialization logic remains separate from the domain model.

---

Avoid Pattern Matching when:

- Dynamic dispatch already solves the problem.
- The hierarchy is expected to grow frequently.
- A simple method override is sufficient.
- The switch becomes excessively large.

A useful rule:

> **If the behavior naturally belongs to the object, use polymorphism. If the behavior belongs to another subsystem, Pattern Matching is often appropriate.**

Because if the left side fails,

the pattern variable is never created,

# Chapter Summary

Pattern Matching modernizes Java by reducing boilerplate while preserving strong type safety. It began with Pattern Matching for `instanceof` in Java 16 and expanded in Java 21 with Pattern Matching for `switch`, Guarded Patterns, and Record Patterns.

Rather than repeatedly checking types and performing manual casts, developers can express type checks, variable extraction, and conditional logic declaratively. Combined with Sealed Classes, Pattern Matching enables compile-time exhaustiveness checking, making code safer and easier to maintain.

Although Pattern Matching does not improve runtime performance, it significantly enhances readability, reduces errors, and simplifies working with immutable data structures such as Records. In modern Java 21 applications, it is particularly valuable for validation, event processing, DTO mapping, serialization, and other cross-cutting concerns.

A key design principle is to continue using polymorphism when behavior naturally belongs to the object, and to use Pattern Matching when behavior belongs outside the domain model. Understanding both approaches—and when to choose each—is an essential skill for Java backend engineers.

# Introduction

The `switch` statement has been part of Java since its earliest versions.

For many years, it served as a convenient alternative to long chains of `if-else` statements.

Example:

```java
switch(day) {

    case MONDAY:
        System.out.println("Start of Week");
        break;

    case TUESDAY:
        System.out.println("Working Day");
        break;

    default:
        System.out.println("Unknown");
}
```

Although widely used, the traditional `switch` statement had several design limitations that became more apparent as Java evolved.

---

# The Problems with Traditional Switch

The old `switch` suffered from multiple issues.

## Problem 1 – Fall-through

Consider:

```java
int day = 1;

switch(day){

    case 1:
        System.out.println("Monday");

    case 2:
        System.out.println("Tuesday");

    case 3:
        System.out.println("Wednesday");

}
```

Output

```
Monday
Tuesday
Wednesday
```

Most beginners expect only:

```
Monday
```

Instead,

execution continues into every subsequent case.

This behavior is called **fall-through**.

---

# Execution Flow

```
Match Case

      │

      ▼

Execute Code

      │

No break?

      │

      ▼

Continue to Next Case

      │

Continue Again

      │

Continue Again
```

The JVM keeps executing until it encounters:

```
break
```

or reaches the end of the switch block.

---

# Why Was Fall-through Introduced?

Historically,

fall-through was designed to allow multiple cases to share the same logic.

Example:

```java
switch(day){

    case SATURDAY:

    case SUNDAY:

        System.out.println("Weekend");

        break;
```

Both cases intentionally execute the same block.

Although useful,

accidental fall-through became one of the most common sources of bugs in Java.

# Problem 2 – Forgetting `break`

Suppose:

```java
switch(status){

    case SUCCESS:

        sendEmail();

    case FAILURE:

        rollback();

}
```

If the developer forgets:

```java
break;
```

both operations execute.

Instead of:

```
SUCCESS
```

the program performs:

```
SUCCESS

↓

FAILURE
```

This is a classic interview question.

---

# Problem 3 – Switch Could Not Return Values

Suppose we want to assign a value.

Traditional Java required:

```java
String result;

switch(day){

    case MONDAY:

        result = "Start";

        break;

    case FRIDAY:

        result = "Weekend Soon";

        break;

    default:

        result = "Unknown";

}
```

Notice:

- Extra variable
- Repeated assignments
- Verbose code

---

# Problem 4 – Boilerplate

The traditional switch required:

- Variable declaration
- Multiple assignments
- Multiple `break` statements
- Risk of fall-through

Large applications often contained thousands of lines of repetitive switch code.

---

# Java's Goal

Java wanted a switch that was:

- More concise
- Safer
- Expression-oriented
- Less error-prone
- Easier to read

This led to **Switch Expressions**.

# What is a Switch Expression?

Unlike a traditional switch statement,

a Switch Expression produces a value.

Example:

```java

    case MONDAY -> "Start";

    case FRIDAY -> "Weekend Soon";

    default -> "Unknown";

};
```

Notice:

The switch directly returns a value.

No temporary variable is needed.

# Statement vs Expression

This distinction is one of the most frequently asked interview questions.

A statement performs an action.

Example:


It does not produce a value.

Example:

```java

```

returns:

```
30
```

Similarly,

```java

...
}
```

can now evaluate to a value.

```java
        grade = "A";

    default:

        grade = "F";

}
```

---

# Switch Expression

```java
    default -> "F";

```

Notice:

No repeated assignments.

Cleaner.

Safer.

```java
    ...

    break;
```

Modern syntax:

```java

    ...
```

The arrow guarantees:

- No accidental fall-through
- No `break`
- One execution path

```
↓

↓

Need break?

↓

Continue?
```

Modern

```

↓

Execute

↓

Exit
```

Execution automatically ends after the selected case.

Multiple cases can now share the same logic.

Example:

```java
         SUNDAY ->

            "Weekday";

```

Equivalent to the old style:


---

# Benefits

Switch Expressions provide:

- Less boilerplate
- Readable syntax
- Direct value return
- Better compiler analysis
- Easier maintenance

- Prefer Switch Expressions when computing values.

# Introduction

In Part 1, we learned that a Switch Expression returns a value.

Simple example:

```java
String result =

switch(day){

    case MONDAY -> "Start";

    case FRIDAY -> "Weekend Soon";

    default -> "Unknown";

};
```

This works because each case returns **exactly one expression**.

But what if a case requires multiple statements?

For example:

- Logging
- Database call
- Variable calculation
- Method invocation
- Validation

How can a switch expression still return a value?

This is where **`yield`** comes in.

---

# The Problem

Suppose we write:

```java
String result =

switch(day){

    case MONDAY -> {

        System.out.println("Start");

        "Monday";

    }

    default -> "Unknown";

};
```

Compilation Error.

---

# Why?

A block:

```java
{

}
```

contains **statements**.

A Switch Expression requires **one value** to be returned.

Simply writing:

```java
"Monday";
```

does not return anything.

The compiler needs an explicit instruction.

---

# The Solution — `yield`

Java introduced the `yield` keyword.

Example:

```java
String result =

switch(day){

    case MONDAY -> {

        System.out.println("Start");

        yield "Monday";

    }

    default -> "Unknown";

};
```

Output:

```
Start
```

Variable:

```java
result
```

contains:

```
Monday
```

---

# What Does `yield` Mean?

Think of it as:

```
Return

FROM THIS SWITCH CASE
```

Unlike:

```java
return;
```

which exits an entire method,

`yield` exits only the current switch expression and provides its value.

---

# `return` vs `yield`

Example:

```java
public String test(){

    return "Java";

}
```

Execution:

```
Method

↓

Return

↓

Method Ends
```

---

Now:

```java
switch(day){

    case MONDAY -> {

        yield "Start";

    }

}
```

Execution:

```
Switch

↓

Yield

↓

Continue Method
```

The enclosing method keeps running after the switch expression has produced its value.

---

# Example

```java
String message =

switch(day){

    case MONDAY -> {

        System.out.println("Working");

        yield "Start";

    }

    default ->

            "Other";

};

System.out.println(message);
```

Output:

```
Working
Start
```

The method continues after the switch.

---

# When is `yield` Required?

### Single Expression

```java
case MONDAY -> "Start";
```

No `yield`.

---

### Multiple Statements

```java
case MONDAY -> {

    log();

    validate();

    yield "Start";

}
```

`yield` is mandatory.

---

# Visual Rule

```
Single Expression

↓

No yield
```

```
Block

↓

yield Required
```

---

# Another Example

```java
int number =

switch(day){

    case MONDAY -> {

        int x = 10;

        int y = 20;

        yield x + y;

    }

    default -> 0;

};
```

Output

```
30
```

---

# Can `yield` Return Any Type?

Yes.

The returned value depends on the switch expression type.

Example

```java
String result =
```

Every branch must produce a `String`.

---

Another example

```java
int value =
```

Every branch must produce an `int`.

---

# Type Consistency

Correct:

```java
String result =

switch(day){

    case MONDAY ->

            "Start";

    default ->

            "Unknown";

};
```

---

Incorrect

```java
String result =

switch(day){

    case MONDAY ->

            "Start";

    default ->

            100;

};
```

Compilation Error.

The compiler attempts to determine a common type for the expression. Here, one branch produces a `String` and another an `int`, so the switch expression is not type-consistent.

---

# Type Inference

Suppose:

```java
var result =

switch(day){

    case MONDAY ->

            "Start";

    default ->

            "Unknown";

};
```

Compiler infers:

```
String
```

because every branch returns a `String`.

---

# Exhaustiveness

One of the most important features.

Suppose:

```java
enum Day{

    MONDAY,

    TUESDAY

}
```

Switch:

```java
String result =

switch(day){

    case MONDAY ->

            "Start";

};
```

Compilation Error.

---

# Why?

Switch Expressions **must** always produce a value.

What happens if:

```
day == TUESDAY
```

?

There is no value to return.

---

# Solution

Handle every possibility.

```java
switch(day){

    case MONDAY ->

            "Start";

    case TUESDAY ->

            "Work";

}
```

Or:

```java
default ->

        "Unknown";
```

---

# Exhaustiveness vs Statements

Traditional switch:

```java
switch(day){

    case MONDAY:

        System.out.println();

}
```

This compiles.

---

Switch Expression:

```java
String x =

switch(day){

    case MONDAY ->

            "Start";

};
```

Compilation Error.

Expressions **must** be exhaustive because they are required to produce a value.

---

# Why is This Safer?

Imagine:

```java
String result =

switch(day){

...
};
```

If one path returns nothing,

the variable would remain uninitialized.

The compiler prevents this entire class of bugs.

---

# Arrow Syntax vs Colon Syntax

Modern Java still supports both.

---

## Colon Syntax

```java
switch(day){

    case MONDAY:

        yield "Start";

    default:

        yield "Unknown";

}
```

Notice:

Even with colon syntax,

`yield` is still required inside a Switch Expression.

---

## Arrow Syntax

```java
switch(day){

    case MONDAY ->

            "Start";

    default ->

            "Unknown";

}
```

Cleaner.

Preferred.

---

# Can We Mix Them?

Technically,

Java allows a switch expression to use colon labels with `yield` or arrow labels consistently within the language rules, but **mixing styles within the same switch is strongly discouraged** because it hurts readability.

Best practice:

Choose **one style** for a given switch expression.

---

# Compiler Internals

Suppose:

```java
String result =

switch(day){

    case MONDAY ->

            "Start";

    default ->

            "Other";

};
```

Compiler conceptually transforms it into something like:

```java
String result;

switch(day){

    case MONDAY:

        result = "Start";

        break;

    default:

        result = "Other";

}
```

The compiler ensures:

- Every path assigns a value.
- No path leaves the variable uninitialized.
- The expression is type-consistent.

---

# Bytecode

Switch Expressions still compile into familiar JVM instructions.

Depending on the switched value,

the JVM may use:

```
tableswitch
```

or

```
lookupswitch
```

The expression semantics are enforced by the compiler, while the JVM executes the generated bytecode.

---

# `tableswitch` vs `lookupswitch`

The JVM chooses between two switch bytecode instructions.

---

### `tableswitch`

Used when values are dense.

Example:

```java
1

2

3

4

5
```

The JVM creates a jump table.

Fast lookup.

---

### `lookupswitch`

Used when values are sparse.

Example:

```java
1

100

1000

9000
```

The JVM builds a lookup structure rather than a dense table.

---

# Does Switch Expression Change Performance?

No significant difference.

The compiler still generates efficient switch bytecode.

The advantages are:

- Readability
- Type safety
- Less boilerplate

rather than raw execution speed.

---

# Best Practices

- Prefer arrow syntax (`->`) for new code.
- Use `yield` only inside block cases.
- Ensure every branch returns a compatible type.
- Keep switch branches concise.
- Let the compiler enforce exhaustiveness.

---

# Common Mistakes

## Mistake 1

Forgetting `yield`.

```java
case MONDAY -> {

    "Start";

}
```

Compilation Error.

---

## Mistake 2

Returning incompatible types.

```java
case MONDAY -> "Start";

default -> 1;
```

Compilation Error.

---

## Mistake 3

Leaving a switch expression non-exhaustive.

```java
switch(day){

    case MONDAY -> "Start";

}
```

Compilation Error.

---

## Mistake 4

Using `return` instead of `yield` inside a block when the intent is to return only from the switch expression.

Remember:

- `yield` exits the switch expression.
- `return` exits the entire method.

---

# Key Takeaways

- `yield` returns a value from a block within a switch expression.
- Single-expression branches do not require `yield`.
- Switch Expressions must be exhaustive.
- Every branch must produce a compatible type.
- The compiler guarantees definite assignment and type safety.
- The JVM continues to use `tableswitch` and `lookupswitch` bytecode instructions.

# Switch Expressions with Enums

Enums are the most common type used with Switch Expressions.

Suppose:

```java
enum OrderStatus {

    SHIPPED,
    DELIVERED
}
```

Traditional switch:

```java
String message;

switch(status) {

    case PAID:

        message = "Payment Received";
        break;

    default:

        message = "Unknown";
}
```

---

# Modern Switch Expression

```java
String message =

switch (status) {

    case CREATED -> "Order Created";

    case PAID -> "Payment Received";

    case SHIPPED -> "Order Shipped";

    case DELIVERED -> "Delivered";

};
```

Notice:

- No `break`
- No temporary variable
- No repeated assignment

---

# Exhaustiveness with Enums

Consider:

```java
enum Day {

    MONDAY,
    TUESDAY,
    WEDNESDAY
}
```

Switch Expression:

```java
String result =

switch(day) {

    case MONDAY -> "M";

    case TUESDAY -> "T";

};
```

Compilation Error.

---

# Why?

The compiler knows:

```
MONDAY

TUESDAY

WEDNESDAY
```

One value is missing.

Since the switch expression must always return a value,

Java rejects the code.

---

# Complete Version

```java
String result =

switch(day){

    case MONDAY -> "M";

    case TUESDAY -> "T";

    case WEDNESDAY -> "W";

};
```

No `default` is required.

---

# Why Avoid `default` for Enums?

One of the most important interview topics.

Suppose today:

```java
enum Status {

    ACTIVE,

    INACTIVE
}
```

Switch:

```java
switch(status){

    case ACTIVE -> "A";

    case INACTIVE -> "I";
}
```

Everything is handled.

---

Six months later:

```java
enum Status {

    ACTIVE,

    INACTIVE,

    SUSPENDED
}
```

Now the compiler immediately reports:

```
Switch is not exhaustive
```

This is a good thing.

It forces developers to update every switch that depends on the enum.

---

# What if We Had Used `default`?

```java
switch(status){

    case ACTIVE -> "A";

    case INACTIVE -> "I";

    default -> "Unknown";
}
```

Now adding:

```
SUSPENDED
```

does **not** produce a compilation error.

Instead,

the new value silently goes to:

```
default
```

This can hide bugs.

---

# Best Practice

For **closed sets** like enums,

prefer handling every constant explicitly.

Avoid `default` unless you genuinely want unknown values to be treated the same.

# How the JVM Executes a Switch

One of the most common senior interview questions is:

> **How does the JVM execute a `switch` statement or switch expression internally?**

Many developers think Java checks every case one by one.

For example:

```java
switch(day){

    case 1 -> ...

    case 2 -> ...

    case 3 -> ...
}
```

Some assume the JVM performs:

```
day == 1 ?

↓

day == 2 ?

↓

day == 3 ?
```

This is **not** how integer switches are typically implemented.

Instead, the compiler generates specialized JVM bytecode.

---

# The Two Switch Instructions

The JVM has only two bytecode instructions for switch operations:

```
tableswitch
```

and

```
lookupswitch
```

The Java compiler decides which one to generate based on the case labels.

---

# `tableswitch`

Used when values are **dense**.

Example:

```java
switch(number){

    case 1 -> ...

    case 2 -> ...

    case 3 -> ...

    case 4 -> ...

    case 5 -> ...
}
```

Notice:

```
1

2

3

4

5
```

No large gaps.

The compiler generates:

```
tableswitch
```

---

# Internal Representation

Conceptually:

```
Index

↓

Jump Table

↓

Target Instruction
```

Example:

```
0 → Case1

1 → Case2

2 → Case3

3 → Case4

4 → Case5
```

The JVM performs a direct indexed jump.

Time complexity is effectively:

```
O(1)
```

---

# Visualization

```
Input = 3

      │

      ▼

Jump Table

      │

      ▼

Case 3
```

No sequential comparisons are required.

---

# `lookupswitch`

Used when values are **sparse**.

Example:

```java
switch(number){

    case 1 -> ...

    case 100 -> ...

    case 500 -> ...

    case 1000 -> ...
}
```

Notice the gaps.

A jump table would waste memory.

The compiler instead generates:

```
lookupswitch
```

---

# Internal Representation

Conceptually:

```
Key

↓

Lookup Structure

↓

Matching Target
```

Example:

```
1      → A

100    → B

500    → C

1000   → D
```

The JVM searches the mapping efficiently.

Time complexity is typically close to:

```
O(log n)
```

depending on the implementation.

---

# How Does the Compiler Decide?

The compiler considers:

- Minimum case value
- Maximum case value
- Number of cases
- Density of the values

Rule of thumb:

```
Dense values

↓

tableswitch
```

```
Sparse values

↓

lookupswitch
```

---

# String Switch Internals

Another common interview question:

> **How can Java switch on Strings when the JVM only understands numeric switch instructions?**

The JVM does **not** have a string switch instruction.

The compiler rewrites the code.

Example:

```java
switch(role){

    case "ADMIN" -> ...

    case "USER" -> ...

}
```

Conceptually becomes:

# Interview Questions & Answers

## Q1. What is a Switch Expression?

### Answer

A Switch Expression is a `switch` that evaluates to a value, allowing direct assignment without temporary variables.

---

## Q2. Which Java version finalized Switch Expressions?

### Answer

Java 14.

JEP 361.

---

## Q3. What problem do Switch Expressions solve?

### Answer

They reduce boilerplate, eliminate accidental fall-through with arrow syntax, and allow `switch` to produce a value directly.

---

## Q4. What is the purpose of `yield`?

### Answer

`yield` returns a value from a block inside a switch expression.

Unlike `return`, it exits only the switch expression, not the enclosing method.

---

## Q5. When is `yield` required?

### Answer

When a switch branch uses a block (`{ ... }`) and needs to produce a value.

Single-expression branches do not require `yield`.

---

## Q6. What is exhaustiveness checking?

### Answer

The compiler verifies that every possible input is handled so the switch expression always produces a value.

---

## Q7. What is the difference between `break` and `yield`?

| `break` | `yield` |
|----------|----------|
| Exits a switch statement | Produces a value from a switch expression |
| Does not return a value | Returns a value to the expression |
| Used in traditional switch | Used in switch expressions |

---

## Q8. What bytecode instructions does the JVM use?

### Answer

- `tableswitch`
- `lookupswitch`

---

## Q9. When does the compiler generate `tableswitch`?

### Answer

When the case values are dense and closely packed.

---

## Q10. When does the compiler generate `lookupswitch`?

### Answer

When the case values are sparse.

---

## Q11. How does a String switch work?

### Answer

The compiler rewrites it using:

1. `hashCode()`
2. Switch on the hash
3. `equals()` to verify the correct string

---

## Q12. Do Switch Expressions improve runtime performance?

### Answer

Not directly.

The benefits are readability, maintainability, and compile-time safety.

---

## Q13. Should Switch Expressions replace polymorphism?

### Answer

No.

Polymorphism is preferred when behavior naturally belongs to the object.

Switch Expressions are better for mapping, validation, serialization, and other external behaviors.

---

## Q14. Are Switch Expressions immutable?

### Answer

The expression itself computes a value. Whether the resulting object is mutable depends on the type being returned.

---

## Q15. Can Switch Expressions be used with Pattern Matching?

### Answer

Yes.

Java 21 combines Switch Expressions with Pattern Matching, Record Patterns, Guarded Patterns, and Sealed Classes for expressive and type-safe code.

---

# Senior Interview Scenario

### Question

Your application has a sealed hierarchy of command objects. Should you implement behavior using polymorphism or a large switch expression?

### Answer

It depends on where the behavior belongs.

- If executing the command is intrinsic to the command itself, prefer polymorphism.
- If multiple external systems (logging, auditing, serialization, validation, reporting) need different processing of the same commands, a switch expression over a sealed hierarchy is often cleaner and benefits from exhaustiveness checking.

---

# Quick Revision Notes

- Switch Expressions became final in Java 14.
- They evaluate to a value.
- Arrow syntax (`->`) removes accidental fall-through.
- `yield` is required for block branches.
- Every switch expression must be exhaustive.
- JVM uses `tableswitch` or `lookupswitch`.
- String switches compile to `hashCode()` + `equals()`.
- Prefer explicit enum handling over unnecessary `default` branches.
- Use polymorphism for intrinsic behavior and switch expressions for external mappings.

# Chapter Summary

Switch Expressions transform Java's traditional `switch` into a modern, expression-oriented language construct. By allowing `switch` to return values directly, introducing arrow syntax to eliminate accidental fall-through, and requiring exhaustiveness, they significantly reduce boilerplate while improving safety and readability.

Internally, the JVM continues to rely on the efficient `tableswitch` and `lookupswitch` bytecode instructions, while the compiler enforces type consistency and complete handling of all possible cases. Combined with Pattern Matching, Records, and Sealed Classes, Switch Expressions become a powerful tool for implementing concise, type-safe logic over finite sets of values.

The key design guideline is to use Switch Expressions when computing values or performing external mappings, while continuing to rely on polymorphism when behavior naturally belongs to the objects themselves. Mastering this distinction is essential for writing clean, maintainable, and modern Java applications.

```java
"""
```

```java
""";
```


```java
"""
```

# Introduction

The most misunderstood aspect of Text Blocks is **indentation**.

Many developers assume Java preserves every space exactly as written.

That is **not** true.

Java automatically removes certain spaces before creating the final `String`.

Understanding **which spaces are removed** and **which are preserved** is one of the most common interview topics.

---

# Two Types of Whitespace

Java divides whitespace into two categories:

```
Whitespace

      │

      ├──────────────┐

      ▼              ▼

Incidental      Essential
Whitespace      Whitespace
```

---

# Incidental Whitespace

Incidental whitespace exists only because the Java code itself is indented.

Example:

```java
public class Main {

    public static void main(String[] args) {

        String text = """

            Hello
            Java

            """;
    }
}
```

Notice:

The spaces before:

```
Hello
Java
```

exist because the code is inside a method.

They are **not** intended to become part of the actual string.

---

# Essential Whitespace

Essential whitespace is intentionally part of the text.

Example:

```java
String text = """

    Name: Guru
      Age: 25

    """;
```

The additional spaces before:

```
Age
```

may be intentional for formatting.

Java preserves them.

---

# Why Remove Incidental Whitespace?

Suppose Java preserved every leading space.

Then moving a method deeper into the code would change the string contents.

Example:

```java
class A {

    String text = """

        Hello

        """;
}
```

Later:

```java
class A {

    void method() {

        if(true){

            String text = """

                Hello

                """;
        }
    }
}
```

Without automatic indentation handling,

the resulting strings would be different.

That would make refactoring dangerous.

---

# Java's Solution

The compiler automatically removes **common leading indentation**.

This process is called:

```
Incidental Whitespace Removal
```

---

# Example

Source:

```java
String text = """

        Java
        Spring
        Boot

        """;
```

Visually:

```
□□□□□□□□Java
□□□□□□□□Spring
□□□□□□□□Boot
```

(Each □ represents one space.)

---

# Compiler Analysis

The compiler examines all **non-blank lines**.

```
Java

Spring

Boot
```

Each begins with:

```
8 spaces
```

Minimum indentation:

```
8
```

---

# Final Result

The compiler removes those 8 spaces.

Runtime string:

```

Spring
Boot
```

---

# Visualization

Source Code

```
□□□□□□□□Java

□□□□□□□□Spring

□□□□□□□□Boot
```

↓

Remove 8 Spaces

↓

Runtime String

```

Spring

Boot
```

# Different Indentation Levels

Example:

```java

        Boot
```

No.

At runtime,

# JVM Internals

One of the most common interview questions is:

> **Does the JVM have a special Text Block object?**

Answer:

**No.**

Text Blocks are purely a **language feature**.

The compiler converts them into ordinary `String` literals before the JVM executes the program.

---

# Compilation Process

Suppose we write:

```java
String query = """

SELECT *
FROM users

""";
```

The compiler performs several steps.

```
Source Code

↓

Read Text Block

↓

Remove Incidental Indentation

↓

Process Escape Sequences

↓

Generate String Literal

↓

Store in Constant Pool
```

After compilation,

the JVM no longer knows that the string originated from a Text Block.

---

# Bytecode Perspective

Conceptually,

the compiler transforms:

```java
String message = """

Hello

""";
```

into something equivalent to:

```java
String message = "Hello\n";
```

The generated bytecode loads a standard string constant.

Typical bytecode instructions include:

```text
ldc

astore
```

`ldc`

means:

```
Load Constant
```

# String Constant Pool

Another favorite interview topic.

Question:

Yes.

Exactly like ordinary string literals.

Example:

```java

""";

""";
```

```

     │

     ▼

 ▲     ▲

 │     │
```

# Heap vs String Pool

```

↓

Objects
```

```
String Pool

↓

```

Text Blocks become string literals,

No.

Example:

```java

""";

""";
```

Only one pooled string literal is created.

Both variables reference it.

# Runtime Creation

Suppose:

```java

""".formatted(name);
```

The template itself is pooled,

but the **result of `formatted()`** is a newly created `String` object.

Diagram:

```

↓

String Pool

↓

formatted()

↓

New Heap String
```

This is the same behavior as other string operations.

# Compiler Responsibilities

The Java compiler is responsible for:

- Parsing the Text Block.
- Removing incidental indentation.
- Processing escape sequences.
- Creating the final string literal.
- Emitting normal bytecode.

---

# JVM Responsibilities

The JVM:

- Loads the compiled string constant.
- Places literals into the String Pool.
- Allocates new strings when runtime operations (such as `formatted()`) produce them.
- Executes the generated bytecode.

The JVM has **no special Text Block execution logic**.

Different tools,

# Interview Questions & Answers

## Q1. What is a Text Block?

### Answer

A Text Block is a multi-line string literal introduced in Java 15 that uses triple double quotes (`"""`) and compiles into a standard immutable `String`.

---

## Q2. Which Java version finalized Text Blocks?

### Answer

Java 15.

JEP 378.

---

## Q3. Why were Text Blocks introduced?

### Answer

To simplify writing multi-line strings such as SQL, JSON, XML, HTML, and GraphQL while reducing escaping and string concatenation.

---

## Q4. Do Text Blocks create a new runtime type?

### Answer

No.

They compile into ordinary `String` objects.

---

## Q5. What is incidental whitespace?

### Answer

Leading indentation that exists because of Java source formatting.

The compiler removes the common incidental indentation from non-blank lines.

---

## Q6. What is essential whitespace?

### Answer

Whitespace intentionally included as part of the text.

It is preserved.

---

## Q7. Does the JVM know about Text Blocks?

### Answer

No.

The compiler converts Text Blocks into ordinary string literals before runtime.

---

## Q8. Are Text Blocks stored in the String Pool?

### Answer

Yes.

Like all string literals, they participate in normal String Pool behavior.

---

## Q9. What is the purpose of `\s`?

### Answer

It represents a space character, particularly useful for preserving trailing spaces that would otherwise be difficult to express or notice.

---

## Q10. What does a trailing `\` do in a Text Block?

### Answer

It suppresses the newline that would normally occur at the end of that source line.

---

## Q11. Can Text Blocks contain quotes without escaping?

### Answer

Yes.

Ordinary double quotes generally do not require escaping inside a Text Block.

---

## Q12. Are Text Blocks mutable?

### Answer

No.

They are ordinary immutable `String` objects.

---

## Q13. Do Text Blocks improve runtime performance?

### Answer

No.

Their benefits are readability, maintainability, and developer productivity.

---

## Q14. Should Text Blocks replace `StringBuilder`?

### Answer

No.

`StringBuilder` is for dynamic string construction, while Text Blocks are for static multi-line literals.

---

## Q15. When should Text Blocks be used?

### Answer

Whenever you need readable multi-line literals such as:

- SQL
- JPQL
- JSON
- XML
- HTML
- GraphQL
- Email templates

# Chapter Summary

Text Blocks modernize the way Java represents multi-line string literals. Instead of relying on repetitive concatenation, escaped quotes, and explicit newline characters, developers can now write text in a form that closely resembles its final output. This significantly improves readability and maintainability, especially for embedded languages such as SQL, JSON, XML, HTML, and GraphQL.

During compilation, the Java compiler removes incidental indentation, processes escape sequences, and generates an ordinary immutable `String` literal. The JVM has no special support for Text Blocks; they behave exactly like traditional string literals at runtime, including participation in the String Pool.

Text Blocks are best viewed as a developer productivity feature rather than a performance optimization. Used appropriately, they make backend code cleaner, reduce formatting errors, and improve the overall maintainability of modern Java applications.
